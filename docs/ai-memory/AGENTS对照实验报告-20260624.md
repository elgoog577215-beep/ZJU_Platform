# AGENTS.md 对照实验报告

生成时间：2026-06-24

## 1. 实验目的

本轮实验的目标不是证明 `AGENTS.md` 写得“更有格式”，而是验证它是否真的改变 Codex 的协作行为：

- 是否更能读出用户真实意图。
- 是否能补充用户没说出的关键认知。
- 是否能把认知落回项目、产品、代码或任务中定位问题。
- 是否能形成更可执行、更可验证的方案。
- 是否会带来过度展开、事实过度推断或表达成本上升。

本轮重点观察的是：`AGENTS.md` 中的“意图空间、认知空间、问题空间、资料空间、方案空间”是否真的比普通回答更稳定、更符合用户意图。

## 2. 实验设计

### 2.1 对照组设置

本轮采用三个条件：

| 条件 | 约束 | 观察目标 |
| --- | --- | --- |
| A 组：不使用 AGENTS.md | 不读取 `AGENTS.md`，不读取项目记忆，不用工具，只按普通 Codex 判断回答 | 观察模型自然能力 |
| B 组：严格使用 AGENTS.md | 只依据 `AGENTS.md` 摘要回答，显式执行空间链路 | 观察规则强制性和完整性 |
| C 组：使用 AGENTS.md 但允许裁剪 | 依据 `AGENTS.md` 摘要回答，但允许根据场景裁剪表达 | 观察真实可用形态 |

### 2.2 评分维度

每个回答按 30 分评估：

| 维度 | 分值 | 说明 |
| --- | ---: | --- |
| 意图识别 | 5 | 是否读出用户真正想解决什么 |
| 认知增量 | 5 | 是否补充了用户没说出的关键框架 |
| 问题落地 | 5 | 是否能落回项目、产品、文件、流程或任务中定位问题 |
| 方案质量 | 5 | 是否有优先级、路径、取舍和验证方式 |
| 表达结构 | 5 | 是否清楚、可读、承接自然 |
| 成本与事实门禁 | 5 | 是否避免过度展开、过度推断和无依据结论 |

## 3. 先导测试：题目过于抽象的问题

一开始做过 5 个偏抽象的问题，用来比较 A/B/C 三组回答。这个先导测试有价值，但不够真实。

### 3.1 发现

- A 组回答自然、简洁，但容易停留在一般经验。
- B 组结构最完整，但有时像“执行模板”。
- C 组通常最接近日常可用回答，但在简单事实题上出现过轻微过度推断。

### 3.2 关键教训

先导测试说明：如果问题太像“考试题”，测出来的是格式控制能力，而不是复杂协作能力。

因此后续改为真实场景题，尤其围绕教育 AI、产品规则、项目页面分析和事实确认。

## 4. 真实场景一：教育 AI 诊断助手质量差

### 4.1 测试问题

> 我们现在有一个在线教育平台里的 AI 诊断助手。学生提交一道编程题后，系统会有题目、标准答案、解析、学生代码、运行结果和错误信息。现在这个 AI 诊断助手效果很差：
>
> - 回答很泛，经常只是说“检查循环条件”“注意变量命名”。
> - 不太区分语法错误、算法思路错误、边界条件错误、复杂度问题。
> - 有时候会直接把完整答案给学生，而不是引导学生自己修。
> - 同一道题多次诊断输出格式不一致，前端不好展示。
> - 日志里只有最终回答，没有题型、知识点、错因标签或诊断过程。
> - 团队有人建议先从 prompt 模板角度优化。
>
> 你先帮我分析：这个 AI 诊断助手质量差，应该从哪里入手优化？请先不要写代码，先给我一个清晰的分析和推进方案。

### 4.2 A 组回答要点

A 组没有使用 `AGENTS.md`，但仍然判断出“不应该把它看成 prompt 问题”。它把问题归纳为：

- 缺少诊断流程。
- 缺少错因分类体系。
- 缺少结构化输出契约。
- 缺少评测闭环。
- Prompt 优化应放在系统结构之后。

推进方案是：

1. 先建立失败样例评测集。
2. 再建立错因分类。
3. 设计诊断流程。
4. 固定输出结构。
5. 最后优化 prompt。

### 4.3 B 组回答要点

B 组显式使用：

1. 意图空间。
2. 认知空间。
3. 问题空间。
4. 方案空间。

它的关键判断是：用户真正要提升的是诊断助手整体质量，不是单纯改 prompt。回答明确区分了：

- 输入材料是否完整。
- 错因分类是否存在。
- 知识点和题型标准是否存在。
- 诊断过程是否稳定。
- 教学策略是否控制“不给完整答案”。
- 输出结构是否能被前端和日志消费。

它把每个现象落回系统问题，例如“日志只有最终回答”对应缺少可观测诊断过程，“输出格式不一致”对应缺少输出契约。

### 4.4 C 组回答要点

C 组保留空间机制，但标题更像真实回答：

- 意图空间：不要把问题缩成 prompt 问题。
- 认知空间：AI 诊断质量由多层链路决定。
- 问题空间：当前真实缺口。
- 方案空间：建议按顺序推进。

C 组的优势是：既让用户看见空间机制，又没有把回答写成规则展示。它的标题本身带判断，不只是空标题。

### 4.5 评价

| 组别 | 得分 | 评价 |
| --- | ---: | --- |
| A | 26/30 | 质量不差，能抓住 prompt 不是根因，但表达结构和问题落地的稳定性略弱 |
| B | 28/30 | 最完整，最能体现规则强制性，但略长，像正式分析报告 |
| C | 29/30 | 最接近真实可用形态，既有空间链路，又有裁剪和判断标题 |

### 4.6 结论

这个场景证明 `AGENTS.md` 对复杂 AI 产品问题有明显作用。它不是让模型“变聪明一点”，而是让模型更稳定地完成：

```text
局部切口 → 真实意图 → 系统认知 → 项目问题 → 统筹方案
```

## 5. 真实场景二：确认提交与推送规则

### 5.1 测试问题

> 我想确认一下现在 AGENTS.md 里到底是“默认自动提交并推送”，还是“默认自动提交但不推送”？你不用改文件，先告诉我你会怎么确认，以及确认后应该怎么跟我说。

### 5.2 A 组回答要点

A 组直接给出了确认路径：

- 只看 `AGENTS.md` 里 Git 相关条款。
- 定位 `默认`、`提交`、`推送`、`用户明确要求`。
- 以 `提交与推送` 专项规则为准。

它还给出了确认后的表达：

> 当前 `AGENTS.md` 的规则是：默认在任务达到收束条件后自动提交，但不推送；只有用户明确要求推送时才推送。

### 5.3 B 组回答要点

B 组先做意图空间判断：用户真正关心的是避免“以为只是提交，结果被推送”的风险。

它给出三种情况：

- 如果原文明确是“默认提交但不推送”，就直接确认。
- 如果原文是“默认提交并推送”，就指出风险。
- 如果原文内部冲突，就按更保守规则处理，并建议统一文档。

### 5.4 C 组回答要点

C 组回答更简洁：

- 读取当前项目根目录 `AGENTS.md`。
- 定位 `提交与推送` 相关段落。
- 检查是否存在全文冲突。
- 确认后说明默认自动提交但不推送。

但它最后加了一句“基于你提供的摘要，当前结论已经指向默认自动提交但不推送”。这句话暴露出一个风险：如果没有读取原文，只凭摘要就下当前结论，容易过度推断。

### 5.5 评价

| 组别 | 得分 | 评价 |
| --- | ---: | --- |
| A | 27/30 | 简洁、任务匹配度高，但仍有一点未读取原文就给确认话术的风险 |
| B | 25/30 | 最稳妥，能处理冲突情况，但对简单事实题略显过度 |
| C | 24/30 | 形式最像真实回复，但出现“凭摘要确认当前规则”的过度推断风险 |

### 5.6 结论

简单事实确认不适合完整展开空间链路。这里最重要的不是认知增量，而是事实门禁：

```text
没有读取原文 → 只能说确认方法，不能说已经确认
```

这说明 `AGENTS.md` 需要进一步强调：空间裁剪只能裁剪篇幅，不能裁剪事实核验和不确定性边界。

## 6. 真实场景三：AI 诊断“不要直接给答案”

### 6.1 测试问题

> 我们平台准备给学生的 AI 诊断增加一个“不要直接给答案”的规则。现在团队有人说，只要在 prompt 里加一句“不要直接给完整答案”就行。我觉得没这么简单。你先帮我从产品和系统设计角度分析一下，这个规则应该怎么设计，才能既不泄露答案，又真的帮助学生继续往下做。

### 6.2 A 组回答要点

A 组判断准确：这不应只是 prompt 里的一句话，而应设计成学习支架策略。

它提出：

- 先定义什么叫泄露答案。
- 使用提示等级，而不是一刀切。
- 系统上不要让模型自由发挥。
- 产品交互要配合。
- 需要例外边界。

回答很实用，但空间承接关系不够显式。

### 6.3 B 组回答要点

B 组用空间链路把问题拆开：

- 意图空间：目标不是禁止 AI 给答案，而是既不替学生完成，又让学生知道下一步怎么做。
- 认知空间：需要定义答案边界、教学递进和系统约束。
- 问题空间：只加 prompt 会出现边界不清、帮助不足、追问绕过、无法验证等问题。
- 方案空间：设计“提示阶梯 + 答案边界 + 输出审查”。

它还提出最小可落地版本：先做红线定义、提示等级、固定输出结构和评测样例。

### 6.4 C 组回答要点

C 组把规则表达成“帮助强度控制规则”，不是单纯“不给答案”。它特别清楚地区分：

- 答案泄露不只是完整代码，也包括最终数值、关键公式、完整步骤、可直接复制的修复代码。
- 不同错误类型需要不同提示梯度。
- 学生追问时也要防绕过。
- 输出结构和系统级审查要配合。

### 6.5 评价

| 组别 | 得分 | 评价 |
| --- | ---: | --- |
| A | 27/30 | 产品感不错，回答自然，但系统化程度略弱 |
| B | 29/30 | 结构完整，能把产品规则、系统约束和验证连起来 |
| C | 30/30 | 最好地平衡了结构、判断和可执行性 |

### 6.6 结论

这个场景说明，`AGENTS.md` 对“用户觉得没这么简单”的问题尤其有效。因为这种问题本身就需要：

```text
识别隐含反对意见 → 建立认知框架 → 找 prompt-only 的真实缺口 → 设计系统规则
```

C 组表现最好，说明复杂问题中最好的输出形态不是裸露模板，而是“空间机制内化后的结构化判断”。

## 7. 真实场景四：活动广场页面混乱

### 7.1 测试问题

> 我觉得我们现在项目里的活动广场页面有点乱，卡片、筛选、搜索、推荐活动、投稿入口都堆在一起。我不想你马上改代码，你先告诉我如果要优化这个页面，你会怎么分析？你会先看哪些文件、看哪些交互、从哪些维度判断问题？

### 7.2 A 组回答要点

A 组把问题看成“信息架构 + 任务流 + 视觉密度”，没有急着改样式。

它会先看：

- 页面入口。
- 组件拆分。
- 数据来源。
- 状态管理。
- 样式文件。
- i18n 文案。

它会检查：

- 浏览活动。
- 搜索。
- 筛选。
- 推荐活动点击。
- 投稿入口。
- 空状态和加载状态。
- 移动端。

回答很实用，但更多是分析清单。

### 7.3 B 组回答要点

B 组先指出：页面问题本质不是“哪里挪一下”，而是活动广场到底服务哪条主路径。

它明确了可能冲突的目标：

- 快速找到想参加的活动。
- 发现平台推荐活动。
- 鼓励用户投稿。
- 展示活动生态丰富度。

它把分析路径分为：

- 信息架构。
- 用户任务流。
- 视觉优先级。
- 页面入口文件。
- 卡片组件。
- 筛选搜索组件。
- 推荐模块。
- 投稿入口。
- 样式与响应式。
- i18n 文案。

### 7.4 C 组回答要点

C 组没有机械使用“意图空间、认知空间、问题空间、方案空间”四个标题，而是转成更自然的结构：

1. 意图判断。
2. 认知维度。
3. 我会先看哪些文件。
4. 我会重点看哪些交互。
5. 最后会形成什么结论。

它的关键判断是：

> 页面大概率应该先确立“活动列表 + 搜索筛选”为主体，把“推荐活动”和“投稿入口”降级为辅助入口，而不是让五类功能在同一层级同时出现。

这个判断非常符合用户真正想要的“先分析，不要马上改代码”。

### 7.5 评价

| 组别 | 得分 | 评价 |
| --- | ---: | --- |
| A | 27/30 | 清单完整，实用性强，但缺少意图牵引和主次判断 |
| B | 29/30 | 结构完整，能解释为什么乱，但略像正式诊断提纲 |
| C | 30/30 | 最符合实际协作：有空间链路、有文件与交互检查、有初步产品判断 |

### 7.6 结论

项目页面分析场景中，最有效的回答不是直接说“我会看哪些文件”，而是：

```text
先判断页面主任务 → 再给分析维度 → 再列文件和交互 → 最后说明会产出什么诊断
```

这正好验证了 `AGENTS.md` 的设计目的：用户给出的“乱”只是局部感受，Codex 应把它转成可定位、可判断、可执行的分析路径。

## 9. 核心结论

### 9.1 `AGENTS.md` 确实有效

在复杂讨论、AI 产品设计、项目页面分析中，使用 `AGENTS.md` 后，回答明显更容易做到：

- 不被用户的局部切口限制。
- 不把 prompt、样式、单点文案误当成根因。
- 主动补充认知框架。
- 把认知落回项目或产品问题。
- 给出更完整的方案和验证路径。

### 9.2 最好的形态是 C 组

严格使用 `AGENTS.md` 的 B 组能保证完整，但有时显得重。

C 组更接近真实可用形态：它保留空间机制，但会根据问题裁剪表达，把标题写成判断句，而不是机械栏目。

最推荐的表达方式是：

```text
空间名 + 本轮关键判断
```

例如：

- 意图空间：不要把问题缩成 prompt 问题。
- 认知空间：AI 诊断质量由系统链路决定。
- 问题空间：当前项目缺的是可观测诊断过程。
- 方案空间：先建立评测集，再改流程和 prompt。

### 9.3 简单事实题需要更强事实门禁

提交与推送规则确认这个场景暴露出一个问题：

如果没有读取原文，不能说“我确认了”。即使空间机制裁剪了，也必须保留事实核验边界。

需要写进 `AGENTS.md` 的关键规则是：

> 裁剪只能裁剪表达篇幅和标题，不能裁剪事实核验、依据说明、不确定性边界和风险门禁。

### 9.4 A 组不差，但稳定性不足

A 组很多回答也不错，这说明基础模型本身有能力。但 `AGENTS.md` 的价值不在于让每次单点回答都压倒普通模型，而在于：

- 稳定触发深层判断。
- 降低漏掉问题空间的概率。
- 让用户能看见回答背后的推理承接。
- 在复杂项目中减少“只改表层”的风险。

## 10. 建议新增或调整的 AGENTS.md 规则

### 10.1 在“表达校准”或“空间裁剪”中加入标题策略

建议加入：

```markdown
复杂问题中，如果显式输出空间标题，标题不应只是“意图空间 / 认知空间 / 问题空间 / 方案空间”的空标签，而应尽量写成“空间名 + 本轮关键判断”。

例如：
- 意图空间：不要把问题缩成 prompt 问题。
- 认知空间：诊断质量由输入、标准、流程、输出和评测共同决定。
- 问题空间：当前项目缺的是稳定诊断过程，不只是提示词。
- 方案空间：先建立评测集，再改造诊断链路。

这样既能让用户看见空间机制，也能避免回复像模板。
```

### 10.2 在“空间裁剪与反误用”中强化事实门禁

建议加入：

```markdown
空间裁剪只能裁剪表达长度、标题数量和展开程度，不能裁剪事实核验、来源依据、不确定性边界和风险门禁。

如果没有读取原文、运行命令、打开页面或核验资料，只能说明“我会如何确认”或“基于当前信息的初步判断”，不得说“我已经确认”。

对文件规则、提交推送、部署状态、线上配置、版本价格、政策规则等事实敏感问题，应优先给出核验路径；在核验前避免把摘要、记忆或推断当作当前事实。
```

### 10.3 在“消息回复阶段”中加入场景化表达模式

建议加入：

```markdown
Codex 应根据场景选择表达形态：

- 简单事实确认：直接回答 + 核验依据或核验路径。通常不需要显式展开空间标题。
- 复杂分析讨论：显式呈现被激活的关键空间，尤其是意图、认知、问题和方案。
- 项目分析但暂不执行：先说明分析对象和判断维度，再说明会读取哪些文件、检查哪些交互、产出什么诊断。
- 用户提出局部方案：先判断该局部方案是不是根因，再把认知扩展落回问题空间。
- 用户明确要执行：先快速补齐缺失空间，再进入动作执行阶段。
```

### 10.4 在“问题空间”中强化项目落地定义

建议补充：

```markdown
问题空间的输出不应只是“有哪些问题”，而应回答：

- 认知空间中提到的关键维度，在当前项目里分别对应哪些文件、模块、数据、交互或流程。
- 当前对象中哪些维度已经存在，哪些缺失，哪些虽存在但不稳定。
- 哪些问题是根因，哪些只是表层表现。
- 哪些问题会直接影响后续方案和验证方式。

如果用户暂时不要求执行，问题空间可以先给“分析路径”；如果用户要求执行，则应进一步读取项目并定位具体文件和证据。
```

## 11. 不建议加入 AGENTS.md 的内容

以下内容不建议直接加入 `AGENTS.md`，否则会让规则变臃肿：

- 每个具体场景的完整回答模板。
- 某个页面、某个教育 AI 模块的固定解决方案。
- 过细的评分表。
- “每次必须输出五个标题”这类硬模板。
- “所有问题都必须深度展开”这类会增加成本的规则。

这些内容更适合保留在项目记忆或报告中，作为后续重构 `AGENTS.md` 的参考。

## 12. 下一步建议

下一步不建议大幅重写 `AGENTS.md`，而是做一次小规模但系统的增强：

1. 强化“空间标题应带判断”的表达规则。
2. 强化“裁剪不等于跳过事实核验”的门禁。
3. 增加“不同场景采用不同表达形态”的调度规则。
4. 细化问题空间如何落回项目文件、模块、数据和交互。

这四项能明显吸收本轮实验的有效结论，同时不会破坏当前 `AGENTS.md` 已经形成的主结构。
