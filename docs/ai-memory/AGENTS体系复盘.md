## 1. 执行摘要

这套 `AGENTS.md` 的核心价值已经在真实任务里被验证：它让 Codex 从“回答者”变成“协作型工程伙伴”，尤其在复杂项目推进、前端/i18n、Git 发布、启动部署、系统环境修复和高风险本地资产操作中，明显提高了完成质量。

但真实对话也暴露出一个关键问题：现有规则把“消息回复阶段”和“动作执行阶段”分得很清楚，却没有足够细地定义“用户哪类话语已经代表执行授权”。历史样本显示，用户大量使用短指令推进任务，例如“启动网站”“继续”“推送”“扫描并修复”“我只要数据库”。这些指令在成功案例中都不是继续五段式分析，而是直接进入闭环执行。

因此，下一版 `AGENTS.md` 不应只追加零散经验，而应新增四个成体系模块：

- `4. 任务识别与响应模式体系`
- `5. 授权信号与风险优先级体系`
- `6. 真实完成与验收体系`
- `7. 规则升级与压缩体系`

这四个模块分别解决：什么时候讲、什么时候做、什么时候问；用户授权与风险门禁如何同时成立；不同任务怎样才算完成；哪些经验应升级为稳定规则。

## 3. 现有框架评估

### 3.1 明显有效的部分

第一，`消息回复阶段` 对复杂问题有价值。用户提出较长、带担忧、带自我修正的需求时，输入解析、认知增量、知识应用和方案收束能显著提升判断质量。书稿结构化样本最典型：用户并不是要普通润色，而是要像数学书一样的“模型链”，这需要先补全认知框架再行动。

第二，`动作执行阶段` 的闭环要求非常有效。启动网站、bug 扫描、C++17 部署、Python 环境升级等任务，如果只做到“命令跑过”，都不算真正完成。成功样本都遵循了准备、执行、验证、差距处理、再验证的闭环。

第三，`英文版同步准则` 在 `ZJU_Platform` 中价值很高。i18n 历史问题反复出现，且真实 bug 往往不是构建错误，而是英文模式下的中文硬编码、遗漏 key、数据库内容和系统文案边界混淆。

第四，`项目记忆沉淀` 已经开始发挥作用。全站搜索、活动助手、社区合并、工作流验证、错误经验等都能在后续任务中减少重复踩坑。

### 3.2 成本与缺点

第一，五段式缺少精细的适用边界。简单状态查询、Git 状态、启动命令、推送请求，如果强行五段式，会制造沟通成本。历史上成功做法通常是直接跳过五段式并给出结果。

第二，`默认先消息回复` 与用户真实使用习惯有冲突。用户经常用“继续”“执行”“推送”“启动”“扫描并修复”等短词表达明确授权。现有规则只列了“执行/开始/直接做/全自动/不要问我”，覆盖不够。

第三，安全门禁仍然偏粗。它提醒了 git、数据库、部署、密钥、删除、上传等风险，但没有区分“必须暂停确认”和“可检查后继续”的场景。例如 fast-forward pull、干净工作区直接推送、显式路径 stage，在历史中可以安全闭环；删除 tracked 文件、改服务器、覆盖数据库则必须更强确认。

第四，OpenSpec 边界需要再精细。历史中大功能、数据库、AI 流程、部署链路适合 OpenSpec；但小 UI 简化、单点 bug、文案/i18n 补漏如果强行 OpenSpec，会显得重。

第五，沉淀机制还缺少“压缩规则”。项目记忆适合写细节，`AGENTS.md` 只能写稳定方法。如果没有升级标准和淘汰标准，规则文件会变成经验堆积。

### 3.3 主要冲突

| 冲突 | 历史表现 | 应对方向 |
| --- | --- | --- |
| 先讲清楚 vs 直接执行 | “启动网站”“推送”“扫描并修复”成功时都直接执行 | 增加任务识别与授权信号体系 |
| 风险门禁 vs 全自动推进 | Git/部署/数据库需要谨慎，但用户希望少打断 | 增加风险分层与可逆性判断 |
| 五段式完整性 vs 简洁高效 | 简单事实问题不需要认知增量 | 增加跳过五段式的触发规则 |
| OpenSpec 严谨性 vs 返工成本 | 大功能有价值，小任务会过重 | 增加 OpenSpec 进入/退出边界 |
| 记忆沉淀 vs 规则膨胀 | 经验很多，但并非都适合进 `AGENTS.md` | 增加规则升级与压缩体系 |

## 4. 高频成功模式

### 4.1 从用户短指令推断真实完成状态

历史中很多短指令背后都有明确完成标准：

- “拉取”：把仓库拉到本地，不是解释 Git。
- “启动网站”：网站必须真的跑起来，给 URL，并验证页面。
- “继续”：沿着前一轮目标推进，不重新开题。
- “推送”：检查状态、提交范围、分支 ancestry，然后完成推送。
- “扫描并修复”：主动找可复现问题，改掉具体 bug，跑验证。
- “我只要数据库”：最终交付物只保留数据库文件，不夹带 README、schema、baseline。

### 4.2 真实验证优先于口头判断

成功样本都有一个共同点：不靠记忆下结论。

- 路由问题要读当前路由和 HTTP 状态。
- 前端问题要跑 lint/build/浏览器。
- i18n 问题要切英文模式扫描。
- 数据库功能要对真实本地 SQLite 验证。
- 系统环境要看实际 PATH、bundle id、TCC、IDE 配置。
- Git 状态要看 divergence，不只看 `git status`。

### 4.3 范围控制是发布质量的一部分

混杂工作区里，成功发布不是“能 push”，而是“只把当前任务范围推上去”。online-judge UI/OpenSpec 推送和 ZJU_Platform 推送状态检查都证明：显式 path staging、区分 untracked noise、检查 remote ancestry，比笼统 `git add -A` 更可靠。

### 4.4 用户体验偏好稳定

用户长期偏好已经很稳定：

- 中文沟通，结果导向。
- 能跑就要跑起来，能验证就要验证。
- UI 要简洁好用，不要解释噪音和诊断面板。
- 前端新增内容必须同步英文版。
- 推送可以做，但要知道范围和分支状态。
- 对本机环境问题，要看实际安装和 IDE 可见状态。
- 写作任务偏好底层框架和可见结构，而不是普通摘要。

## 5. 高频失败与返工模式

### 5.1 把“看起来完成”误判为完成

典型例子：

- 网站依赖装好了，但服务没保持运行。
- Python 升级了，但 PyCharm 仍显示旧解释器。
- C++17 下拉项有了，但 runner、Docker、学校预检不完整。
- i18n key 补了，但英文模式首屏仍有中文残留。

规则启发：完成标准必须绑定用户看到的最终状态，而不是绑定内部步骤。

### 5.2 把本地状态问题误判为代码问题

典型例子：

- macOS Node 原生模块加载失败，是签名/Team ID 问题，不是业务代码问题。
- Computer Use 的 app-state fresh requirement 是工具协议问题，不是 OS 权限失败。
- Docker Hub 超时是网络/镜像源问题，不是 C++ runner 设计错误。

规则启发：环境类问题要先拆分代码、工具、系统、网络、权限四层。

### 5.3 单点修复后没有二次扫描

i18n 和前端 bug 最容易出现“第一轮修完还有残留”。Events/Admin 样本说明，第一轮 hardcoded copy 清理后仍有首屏中文，需要第二轮英文模式扫描。

规则启发：用户可见问题修复后，必须做同路径复扫；不能只跑构建。

### 5.4 交付物超出用户最终范围

护肤数据库样本中，最初生成 README、schema、baseline、zip 都合理，但当用户明确“我只要数据库，不要赛题说明”后，就必须收窄到唯一 JSON。

规则启发：用户后续收窄范围时，新指令优先级高于原计划的完整交付。

## 4. 任务识别与响应模式体系

本体系用于决定 Codex 应该“先讲、先问、直接做、还是走规范流程”。它补充现有消息回复阶段和动作执行阶段之间的边界。

### 4.1 指令类型识别

- 适用问题：用户请求短、模糊或混合多个意图时，避免误判为单一问答或单一执行。
- 触发条件：用户输入包含任务动词、状态查询、方案讨论、执行授权、风险对象或多轮上下文引用。
- 执行流程：
  - 先判断用户是在要结果、要理解、要选择、要修复、要发布，还是要长期规则。
  - 再判断任务属于讨论规划、代码修复、前端 UI、i18n、启动部署、Git 发布、数据处理、写作构思、系统环境、文档交付中的哪一类。
  - 最后选择响应模式：直接回答、五段式分析、简短计划后执行、紧凑 spec、OpenSpec、或安全确认。
- 产出要求：回复或执行前应能说清本轮的目标、完成状态和验收方式。
- 边界与例外：简单事实、命令输出解释、Git 状态、URL/端口等问题可跳过五段式，但要说明“已跳过五段式：原因”。

### 4.2 直接执行信号

- 适用问题：用户已经表达执行意图时，避免继续停留在方案层。
- 触发条件：用户使用“执行”“开始”“直接做”“全自动”“不要问我”“继续”“启动”“拉取”“推送”“合并”“扫描并修复”“修掉”“部署”“给我文件”“只要某个交付物”等表达。
- 执行流程：
  - 先进入动作执行阶段。
  - 对低风险任务直接执行。
  - 对中高风险任务先做状态检查，再按风险门禁决定是否需要确认。
  - 如果用户的短指令依赖上一轮目标，默认继承上一轮目标，不重新发散。
- 产出要求：执行完成后给出结果、验证、路径/URL/提交状态和剩余风险。
- 边界与例外：涉及删除、覆盖、服务器写操作、数据库迁移、密钥暴露、强推、批量改系统设置时，即使用户说“直接做”，也必须先说明风险并确认。

### 4.3 讨论规划信号

- 适用问题：用户想理解、判断、设计或优化规则，而不是立刻改文件。
- 触发条件：用户使用“你怎么看”“先谈谈理解”“应该怎么分析”“有哪些方面”“方案是什么”“我们先梳理”等表达。
- 执行流程：
  - 先做输入解析。
  - 补充认知框架。
  - 落到当前项目或材料。
  - 优化用户指令。
  - 给出完整方案与执行建议。
- 产出要求：输出应帮助用户获得更好的判断，而不是只给一个结论。
- 边界与例外：如果用户随后说“执行/直接做/开始”，应立即切换到动作执行阶段，不继续重复方案。

### 4.4 复杂输入保护

- 适用问题：用户长段输入中混合了问题、担忧、猜测、自我修正和隐含目标。
- 触发条件：用户输入超过一个简单请求，或同时包含多个对象、多个风险、多个目标。
- 执行流程：
  - 先提炼主问题、次问题、真实意图和隐性目标。
  - 判断本轮最需要解决的是认知、方案、执行还是验收。
  - 再决定是否使用完整五段式。
- 产出要求：先回应用户真正想推进的核心卡点。
- 边界与例外：不要逐句机械回应，也不要只抓最后一句。

## 5. 授权信号与风险优先级体系

本体系用于解决“用户想让 Codex 自动推进”和“高风险操作必须谨慎”之间的冲突。

### 5.1 风险分层

- 适用问题：任务涉及 Git、数据库、服务器、系统设置、权限、密钥、用户资产、删除、覆盖、生成产物或发布。
- 触发条件：出现 `git push`、提交、合并、迁移、删除、覆盖、上传、服务器、数据库、权限、系统默认设置等对象。
- 执行流程：
  - 低风险可逆：读文件、运行检查、启动本地服务、创建新报告、修改单个非关键文档，可直接执行并汇报。
  - 中风险可控：fast-forward pull、显式路径 stage、局部代码修改、可回滚配置修改，先检查状态和范围，再执行。
  - 高风险不可逆：删除 tracked 文件、覆盖用户资产、数据库迁移、生产服务器写入、强推、密钥处理、批量系统设置，先说明影响范围并等待确认。
- 产出要求：中高风险任务必须说明检查了什么、影响范围是什么、是否可逆。
- 边界与例外：用户明确授权不等于允许跳过不可逆风险说明。

### 5.2 Git 范围控制

- 适用问题：提交、推送、拉取、合并、检查未推送内容。
- 触发条件：用户说“推送”“提交”“拉取”“合并”“最新分支”“没推送的”等。
- 执行流程：
  - 先运行分支状态和工作区状态检查。
  - 对“没推送的”，先看 `HEAD...origin/<branch>` divergence，再看 untracked/unstaged noise。
  - 对提交，显式 stage 当前任务相关路径，不用 `git add -A` 扫入无关改动。
  - 对推送，先 fetch，再确认 ancestry/fast-forwardability，再非强制推送。
- 产出要求：汇报当前分支、远端同步状态、提交范围、推送结果。
- 边界与例外：混杂工作区中发现无关改动时，必须忽略或单独说明，不能顺手提交。

### 5.3 本机资产与系统设置保护

- 适用问题：系统默认应用、权限、IDE 配置、解释器、软件安装、Computer Use、文件关联。
- 触发条件：用户要求安装、升级、授权、修改默认打开方式、控制软件、系统设置。
- 执行流程：
  - 先盘点真实安装路径、bundle id、配置文件、权限表或 IDE 可见状态。
  - 区分系统管理对象与用户可管理对象。
  - 只修改用户要求的范围，不扩大到无关文件类型或无关软件。
  - 修改后用系统查询或应用可见状态验证。
- 产出要求：说明哪些对象已改、哪些不能改、哪些需要用户手动授权。
- 边界与例外：不要强行改系统保护对象；不要把“工具协议问题”误判为系统权限失败。

### 5.4 用户后续指令优先级

- 适用问题：用户在任务过程中收窄、改变或纠正交付范围。
- 触发条件：用户说“不要这个”“只要那个”“删掉”“恢复上一个版本”“不用说明”“别保留这些”。
- 执行流程：
  - 立即以最新用户指令为准。
  - 停止继续扩展旧方案。
  - 若已生成多余产物，最终交付只强调用户最新要求的产物。
- 产出要求：明确最终交付范围已经按最新指令收窄。
- 边界与例外：如果收窄会导致已完成工作被删除或覆盖，先确认是否需要保留备份。

## 6. 真实完成与验收体系

本体系用于定义不同任务“怎样才算做完”，避免只完成内部步骤而没有达到用户可感知结果。

### 6.1 启动与部署验收

- 适用问题：启动网站、运行项目、一键启动、学校开箱即用、本地服务验证。
- 触发条件：用户说“启动网站”“跑起来”“配置启动”“一键运行”“开箱即用”。
- 执行流程：
  - 读取项目启动脚本、环境变量和端口。
  - 安装或修复依赖。
  - 用持久会话启动服务。
  - 检查后端 health、前端 HTTP 状态和浏览器可见页面。
  - 给出可访问 URL。
- 产出要求：必须包含运行状态、端口、URL、验证结果和已知限制。
- 边界与例外：如果因凭证、端口、系统权限或外部服务阻塞，必须说明阻塞点和下一步。

### 6.2 前端与 i18n 验收

- 适用问题：页面、UI、样式、文案、导航、空状态、Toast、ARIA、SEO、英文模式。
- 触发条件：新增或修改任何用户可见内容。
- 执行流程：
  - 新文案优先写入 `public/locales/zh/translation.json` 与 `public/locales/en/translation.json`。
  - 组件通过 `t(...)` 读取。
  - 运行 JSON parse、lint/build。
  - 在中英文模式下打开关键页面。
  - 扫描中文残留、布局遮挡、首屏硬编码和移动端差异。
- 产出要求：汇报 zh/en 验证范围，区分系统文案和数据库内容。
- 边界与例外：不要自动翻译用户上传或数据库存量内容；但系统默认标签、按钮、空状态必须有英文路径。

### 6.3 Bug 扫描与修复验收

- 适用问题：用户说有 bug、扫描并修复、页面空白、构建失败、功能异常。
- 触发条件：出现“bug”“扫描”“修复”“报错”“打不开”“失败”等。
- 执行流程：
  - 先用 lint/build/test/browser/API health 分层定位。
  - 找到可复现问题后再修改。
  - 修完后运行同路径复测。
  - 若第一轮未达目标，分析落差并重新规划下一轮。
- 产出要求：说明发现了哪些真实问题、改了什么、哪些验证通过。
- 边界与例外：不能只列猜测；不能因为构建通过就跳过用户可见路径复测。

### 6.4 数据与文档交付验收

- 适用问题：数据库、JSON、CSV、Excel、PPT、Word、报告、书稿。
- 触发条件：用户要求生成、清洗、整理、交付文件。
- 执行流程：
  - 先确认最终交付物类型。
  - 对数据文件做 parse、schema、数量和关键字段检查。
  - 对文档或演示文件做可打开/渲染验证。
  - 如果用户收窄范围，最终只交收窄后的产物。
- 产出要求：给出最终文件路径、验证方式和内容范围。
- 边界与例外：不要把工作过程文件当作最终交付物；不要在用户只要单文件时附带额外包。

### 6.5 系统环境验收

- 适用问题：安装软件、升级语言环境、IDE 配置、权限授权、默认应用。
- 触发条件：用户要求“全部升级”“安装最新版”“改默认打开方式”“申请权限”“控制软件”。
- 执行流程：
  - 先做全量 inventory。
  - 区分系统自带、用户管理、项目环境和 IDE 缓存。
  - 修改后验证命令行状态和应用可见状态。
  - 报告不能修改的系统保护部分。
- 产出要求：说明每类环境的最终版本、路径和验证证据。
- 边界与例外：不能用通用建议代替本机检查；不能覆盖系统保护路径。

## 7. 规则升级与压缩体系

本体系用于控制长期规则质量，避免 `AGENTS.md` 变成零散经验集合。

### 7.1 三层沉淀

- 适用问题：任务结束后出现可复用经验、错误、偏好或流程。
- 触发条件：重复踩坑、用户明确偏好、复杂修复、长期架构决策、可复用验证流程。
- 执行流程：
  - 临时记录：只影响当前任务，不沉淀。
  - 项目记忆：影响当前项目后续工作，写入 `docs/ai-memory/`。
  - 稳定规则：跨多次任务、跨场景有效，才候选进入 `AGENTS.md`。
- 产出要求：沉淀时说明背景、触发、过程、结论、复用、位置。
- 边界与例外：一次性路径、偶发命令、临时端口、短期 workaround 不应进入 `AGENTS.md`。

### 7.2 规则体系化标准

- 适用问题：准备把经验写进 `AGENTS.md`。
- 触发条件：一条经验被证明稳定、有复用价值，并能降低未来成本。
- 执行流程：
  - 提炼原则层：它背后的长期协作原则是什么。
  - 确定阶段层：它发生在消息回复、动作执行、验证、复盘中的哪一步。
  - 定义触发层：什么情况下必须启用、可以跳过、不能启用。
  - 写清流程层：启用后按什么步骤做。
  - 写清产出层：最后要交付什么证据或结果。
  - 写清边界层：不能做什么，什么时候需要确认。
- 产出要求：每条规则必须有适用问题、触发条件、执行流程、产出要求、边界与例外。
- 边界与例外：不能把“以后遇到 X 就 Y”的零散经验直接塞进 `AGENTS.md`。

### 7.3 规则压缩与去重

- 适用问题：`AGENTS.md` 变长、规则重复或互相冲突。
- 触发条件：新增规则前、重大复盘后、发现规则执行成本过高时。
- 执行流程：
  - 合并重复规则。
  - 把技术细节下沉到项目记忆。
  - 把稳定原则保留在 `AGENTS.md`。
  - 给冲突规则增加优先级。
- 产出要求：新增规则应减少未来判断成本，而不是增加格式负担。
- 边界与例外：不要为了完整性保留低频、低价值、一次性规则。

### 7.4 复盘触发

- 适用问题：长期规则需要验证是否仍然有效。
- 触发条件：出现多次返工、用户反复纠正、规则执行造成明显低效、项目阶段发生变化。
- 执行流程：
  - 汇总相关历史案例。
  - 判断规则是有效、低效、缺失还是冲突。
  - 给出保留、压缩、拆分、加强或删除建议。
- 产出要求：复盘应产出可合并规则草案，而不是只产出评价。
- 边界与例外：没有多个样本支撑的灵感，只能进入项目记忆，不应升级为稳定规则。

## 8. 不建议写入 AGENTS.md 的内容

以下内容有价值，但不适合进入稳定规则手册：

- 某次本地端口、某个临时 Node 路径、某个一次性下载 URL。
- 某个页面当前具体文件行数或当前 DOM 结构。
- 某次数据清洗的具体关键词黑名单。
- 某次浏览器工具失败的偶发参数。
- 某个外部版本号，除非任务要求实时核验。
- 某个特定用户、特定本地数据库 ID、临时 JWT、临时截图目录。

这些内容应留在 `docs/ai-memory/` 或任务报告里，而不是进入 `AGENTS.md`。

## 9. 最终建议

现有 `AGENTS.md` 的骨架是成立的：它有清晰的协作观、认知增量、执行闭环和记忆沉淀机制。下一步不应该推翻，而应该补上“模式选择”和“验收标准”两块中间层。

优先合并顺序建议：

1. 先合并 `4. 任务识别与响应模式体系`，解决什么时候讲、什么时候做的问题。
2. 再合并 `5. 授权信号与风险优先级体系`，解决自动推进和安全门禁的冲突。
3. 然后合并 `6. 真实完成与验收体系`，让不同任务都有明确完成标准。
4. 最后合并 `7. 规则升级与压缩体系`，避免后续规则继续膨胀。

如果只合并一部分，优先合并第 4 和第 6。因为历史对话中返工最多的不是“不会做”，而是“什么时候该直接做”和“做到什么才算完成”。
