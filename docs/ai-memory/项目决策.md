# 项目决策

## 2026-06-01 全站 AI 搜索与活动推荐助手分工

- 背景：合入 `feat/ai-search-redesign` 的 AI 搜索能力时，需要同时保留 `master` 中基于用户画像和偏好的活动推荐助手。
- 触发：同学分支的搜索能力适合被扩展到活动、AI 社区、影像库，但如果直接替代活动推荐助手，会混淆“主动查找资源”和“根据画像推荐机会”两类产品心智。
- 过程：对比后决定不整包合并目标分支，而是抽取其“自然语言解析、结构化筛选、匹配理由、结果分组”的能力，接入当前主线的数据库、用户系统和现有页面深链。
- 结论：全站 AI 搜索负责跨模块资源发现，入口是 `/api/search` 和全局搜索弹窗；活动推荐助手继续负责基于用户画像、偏好和行为证据的个性化活动推荐，入口保持 `/api/events/assistant`。
- 复用：后续接入治理助手、结构化索引或向量索引时，应优先增强 `globalSearchService` 的召回实现，不改变前端结果协议，也不把搜索 agent 与推荐 agent 合并成一个入口。
- 位置：`server/src/services/globalSearchService.js`、`src/components/SearchPalette.jsx`、`server/src/services/aiAgentRegistryService.js`。

## 2026-06-01 用户系统承载活动推荐画像

- 背景：活动推荐助手原先在卡片内部提供“我的推荐偏好”表单，容易让用户误以为画像只属于活动页，而不是长期账号资料。
- 触发：用户明确要求学院、兴趣、空闲时间等长期信息应在注册或编辑资料时维护，推荐助手只负责读取这些信息做个性化推荐。
- 过程：对比后决定不把推荐助手和搜索助手统一成一个大入口，而是把长期画像沉入用户系统；推荐助手展示画像来源与摘要，并跳转到个人资料设置页维护。
- 结论：`user_event_preferences` 是活动推荐画像的数据表，用户资料设置页是长期维护入口；`EventAssistantPanel` 不再承载画像编辑表单，只读取画像、结合行为证据与本次提问排序。
- 复用：后续新增画像字段时，优先在用户系统资料页和 `userProfileService` 扩展，再让推荐助手消费，不在推荐助手中新增长期资料表单。
- 位置：`src/components/PublicProfile.jsx`、`src/components/EventAssistantPanel.jsx`、`server/src/services/userProfileService.js`。

## 2026-06-02 全站搜索索引先做治理刷新命令

- 背景：全站搜索已经扩展到活动、AI 社区、影像库，下一步需要让治理助手定期生成结构化摘要、关键词和向量兼容索引。
- 触发：如果直接在 Web 进程里启动后台定时写库，会引入多实例重复执行、部署时机、失败重试和线上数据库写入节奏等问题；但如果只停留在 SQL 召回，又无法验证真实库里的智能检索方向。
- 过程：决定先新增 `resource_search_index` 作为统一索引表，并提供 `refresh-resource-search-index.js` 命令。命令负责执行迁移、读取公开已审核资源、生成摘要/关键词/分面/本地轻量向量、记录 `global_search_index` 刷新运行。部署层或 cron 后续可以调用这个命令，而不是让普通请求进程默认定时写库。
- 结论：`globalSearchService` 保留原 SQL 召回，同时合并 `resource_search_index` 补充召回；当前 `vector_json` 是可替换的本地词元哈希向量，不宣称等同真实 embedding/RAG。
- 复用：后续接入模型 embedding、外部向量库或真正的治理定时任务时，优先替换 `resourceSearchIndexService` 的向量生成/检索实现和部署调度，不改变 `/api/search` 的前端结果协议。
- 位置：`server/src/services/resourceSearchIndexService.js`、`server/scripts/refresh-resource-search-index.js`、`server/src/services/globalSearchService.js`。

## 2026-06-02 黑客松成果页采用赛后作品展台结构

- 背景：黑客松成果页原先把全屏吸附、巨大标题、电影框、统计栏、章节导航和大量科技纹理叠在首屏，桌面端标题容易被拆行。
- 触发：用户要求重新思考成果页设计，并在后续明确企业 logo 不应新增到成果页首屏，logo 布局问题属于赛事报名页。
- 过程：决定把成果页从“报名页延长线”改为“赛后作品档案”：标题固定为“AI 全栈极速 / 比赛成果”两行；宣传片先展示封面，点击后播放；赛场照片、优秀作品、合作方按内容价值分区展示。企业 logo 不放在成果页首屏，只在合作方分区作为支持阵容内容出现。
- 结论：`/hackathon` 默认进入成果页，`?view=register` 仍保留报名入口；移动端不再显示第二层页内章节底栏，只保留全站底部导航，避免双导航挤压内容。
- 复用：后续改黑客松页面时，企业 logo 首屏布局只在赛事报名页处理；成果页首屏优先展示赛后影像、作品和成果叙事。
- 位置：`src/components/HackathonShowcase.jsx`、`src/components/HackathonSeasonOne.jsx`、`src/components/HackathonRegistration.jsx`。

## 2026-06-08 白天模式采用纯白底与粉紫点缀

- 背景：博客与播客已经收进 AI 社区后，用户同时反馈白天模式出现过多半渐变按钮、灰底和杂色，整体显得不自然。
- 触发：用户明确偏好 2026 年 3 月到 4 月附近版本里活动页那种少量彩色光晕点缀，同时喜欢粉色、紫色，但不希望全站被灰色或大面积渐变覆盖。
- 过程：对比后决定不继续强化当前绿色、蓝色、灰色混合的日间主题，而是把白天模式的基础层退回纯白和直角感；仅在活动页、AI 社区头部、重点按钮、选中态和辅助光晕里加入低饱和粉紫。
- 结论：白天模式默认背景应接近纯白，卡片使用白色或极浅粉白，按钮避免半渐变，主操作使用实色紫；彩色光晕只作为少量空间点缀，不能喧宾夺主。
- 复用：后续新增白天模式页面或调整组件时，优先使用 `--theme-bg`、`--theme-surface`、`--theme-accent` 等主题变量，避免重新引入大面积灰底、蓝绿渐变或过圆的胶囊按钮。
- 位置：`src/index.css`、`src/components/Events.jsx`、`src/components/AICommunity.jsx`、`src/components/Navbar.jsx`。

## 2026-06-29 历史回填：AI 社区采用统一内容系统

- 背景：AI 社区曾同时存在新闻热榜、技术分享、求助、组队等割裂入口，用户明确希望不要分得太开，而是统一成一个社区内容系统。
- 触发：普通用户也应能投稿新闻热点，所有版面都应进入同一套内容发布/审核/管理逻辑；这不是小 UI 修补，而是信息架构调整。
- 过程：第一版把公开 AI 社区压成统一发帖区概念，四个公共版面为技术分享、求助问答、新闻热点、组队协作；个人投稿管理收口到用户中心 `submissions` tab；旧 `/community/groups` 回到 `/articles`。
- 结论：AI 社区的稳定方向是“统一内容系统 + 分版面语义 + 用户中心投稿管理”，不是右侧独立热榜、独立新闻投稿和独立组队入口并存。
- 复用：后续新增社区版面、审核状态、用户投稿或新闻入口时，应优先接入统一发布/管理路径；旧 `tab=groups` 语义不应复活。
- 位置：`src/components/AICommunity.jsx`、`src/components/PublicProfile.jsx`、`src/components/UserCommunitySubmissions.jsx`、`src/App.jsx`

## 2026-06-29 历史回填：身份归属采用多主体 profile 链接模型

- 背景：活动可能由学生上传，但真实归属又属于学生会、学院、社团等组织；用户希望发布者、发布单位、地点等信息能自动关联，不要制造重复记录。
- 触发：讨论中确认学生是普通用户，官方组织/社团也应有账号；平台历史内容应逐步转到对应认证组织名下。
- 过程：以现有 `profiles`、`profile_members`、`profile_aliases` 为身份锚点，延展 `publisher_profile_id`、`organizer_profile_id` 和别名匹配；地点只作为辅助匹配/筛选信息，不作为主要身份主体。
- 结论：平台内容采用“一条内容记录 + 多主体角色链接”的模型：上传者、发布单位、主办组织、历史归属可以共同指向同一活动，而不是复制多份内容。
- 复用：后续做活动/组织/发布身份/历史迁移时，先确定归属规则，再做数据迁移；不要退回到扁平角色表或手动复制记录。
- 位置：`profiles`、`profile_members`、`profile_aliases`、`publisher_profile_id`、`organizer_profile_id`、`src/components/UploadModal.jsx`

## 2026-06-29 历史回填：历史平台内容迁移为认证组织所有权

- 背景：早期平台可能由平台账号代发组织活动，后续组织/社团有独立账号后，需要让历史内容归属更真实。
- 触发：用户表达“数据过渡到他们的账号名下”的方向，且先要求把规则定下来，再考虑具体迁移。
- 过程：先把目标模型定为 profile-based ownership/attribution reassignment，再考虑 alias matching、人工审核和迁移批次。
- 结论：历史平台代发内容应通过归属/所有权重分配迁入认证组织 profile，而不是删除重建或复制新记录。
- 复用：后续迁移前先输出匹配规则和审计清单；位置、标题、发布单位、别名只能作为证据，不应单独决定高风险迁移。
- 位置：`profiles`、`profile_aliases`、`publisher_profile_id`、`organizer_profile_id`

## 2026-06-29 历史回填：Android 优先复用 WebView/PWA 包装路线

- 背景：用户询问 Android 开发目录和 APK 交付时，项目已有 Web/PWA 基础，不需要一开始做完整原生重写。
- 触发：用户接受“先搞定第一步”，目标是先有可用 Android App，而不是重构成全原生应用。
- 过程：确认 Android 项目在 `android-twa`，稳定关注 `AppWebViewActivity.java`、`LauncherActivity.java`、`AndroidManifest.xml`、`app/build.gradle`；签名 APK 输出到 `output/android/apk/tuotuzju-v8-20260617-signed.apk`，下载副本在 `public/downloads/tuotuzju-android.apk`。
- 结论：当前 Android 方向是复用 Web App 的 WebView/TWA-style wrapper。除非用户明确要求原生能力，否则优先保证包装、签名、下载和 assetlinks 对齐。
- 复用：后续 Android 任务先回答精确目录和 APK 路径；改包时重新核验 package name、versionCode/versionName、签名、下载副本和可安装性。
- 位置：`android-twa/`、`android-twa/app/src/main/java/com/tuotuzju/app/AppWebViewActivity.java`、`public/downloads/tuotuzju-android.apk`

## 2026-06-29 历史回填：全站搜索索引是补充召回，不替代响应协议

- 背景：全站 AI 搜索扩展到活动、AI 社区、影像库后，需要统一索引公开资源并给治理助手/搜索入口使用。
- 触发：直接在 Web 进程里做后台定时写库会带来多实例、部署时机和失败重试问题；但只靠 SQL 召回又不足以验证智能检索方向。
- 过程：建立 `resource_search_index` 和 `refresh-resource-search-index.js` 命令；`globalSearchService` 合并 SQL 召回与索引召回，并按 `type:id` 去重；本地验证 262 个公开资源覆盖率 100%。
- 结论：`resource_search_index` 是补充召回层，`vector_json` 是可替换的轻量 token-hash 向量，不宣称为真实 embedding/RAG；前端 `/api/search` 响应协议保持稳定。
- 复用：后续接入模型 embedding、外部向量库、cron 或治理助手时，替换索引生成/检索和调度，不轻易改变前端结果协议。
- 位置：`server/src/services/resourceSearchIndexService.js`、`server/scripts/refresh-resource-search-index.js`、`server/src/services/globalSearchService.js`

## 2026-06-29 用户系统升级先做账号与身份控制中心

- 背景：用户要求网站用户系统全面迭代升级，现有系统已经有登录、个人资料、个人名片、活动画像、身份认证、组织 profile、投稿和成果认领，但入口分散。
- 触发：如果直接重写登录注册或新增平行账号模型，会绕开既有 `profiles`、`profile_members`、`user_identity_claims` 和投稿/成果链路，破坏之前定下的多主体 profile 方向。
- 过程：第一版选择新增只读聚合接口 `/api/users/me/overview`，复用现有表统计资料完整度、身份认证、可管理主体、内容状态和成果认领；前端在个人中心 owner 视图展示账号与身份控制中心，并把缺失项链接到已有设置页。
- 结论：用户系统的稳定迭代方向是“聚合控制中心 + 既有身份/profile 链路继续加厚”，不是另建账号中心或重写权限模型。资料完整度用于引导用户补全，不作为权限门槛。
- 复用：后续做组织入驻、注册引导、资料完整度、认证身份、历史归属或投稿管理时，优先挂到 `/users/me/overview` 和个人中心控制中心；只有现有表无法表达时再考虑新表。
- 位置：`server/src/controllers/userController.js`、`server/src/routes/api.js`、`src/components/PublicProfile.jsx`、`src/components/ProfilePage.jsx`、`src/components/profile/UserSystemOverview.jsx`、`src/App.jsx`、`public/locales/zh/translation.json`、`public/locales/en/translation.json`
