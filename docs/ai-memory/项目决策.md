# 项目决策

## 2026-06-01 全站 AI 搜索与活动推荐助手分工

- 背景：合入 `feat/ai-search-redesign` 的 AI 搜索能力时，需要同时保留 `master` 中基于用户画像和偏好的活动推荐助手。
- 触发：同学分支的搜索能力适合被扩展到活动、AI 社区、影像库，但如果直接替代活动推荐助手，会混淆“主动查找资源”和“根据画像推荐机会”两类产品心智。
- 过程：对比后决定不整包合并目标分支，而是抽取其“自然语言解析、结构化筛选、匹配理由、结果分组”的能力，接入当前主线的数据库、用户系统和现有页面深链。
- 结论：全站 AI 搜索负责跨模块资源发现，入口是 `/api/search` 和全局搜索弹窗；活动推荐助手继续负责基于用户画像、偏好和行为证据的个性化活动推荐，入口保持 `/api/events/assistant`。
- 复用：后续接入治理助手、结构化索引或向量索引时，应优先增强 `globalSearchService` 的召回实现，不改变前端结果协议，也不把搜索 agent 与推荐 agent 合并成一个入口。
- 位置：`server/src/services/globalSearchService.js`、`src/components/SearchPalette.jsx`、`server/src/services/aiAgentRegistryService.js`。

## 2026-06-01 用户系统承载活动推荐画像

- 背景：活动推荐助手原先在卡片内部提供“我的推荐偏好”表单，容易让用户误以为画像只属于活动页，而不是长期账号资料。
- 触发：用户明确要求学院、兴趣、空闲时间等长期信息应在注册或编辑资料时维护，推荐助手只负责读取这些信息做个性化推荐。
- 过程：对比后决定不把推荐助手和搜索助手统一成一个大入口，而是把长期画像沉入用户系统；推荐助手展示画像来源与摘要，并跳转到个人资料设置页维护。
- 结论：`user_event_preferences` 是活动推荐画像的数据表，用户资料设置页是长期维护入口；`EventAssistantPanel` 不再承载画像编辑表单，只读取画像、结合行为证据与本次提问排序。
- 复用：后续新增画像字段时，优先在用户系统资料页和 `userProfileService` 扩展，再让推荐助手消费，不在推荐助手中新增长期资料表单。
- 位置：`src/components/PublicProfile.jsx`、`src/components/EventAssistantPanel.jsx`、`server/src/services/userProfileService.js`。

## 2026-06-02 全站搜索索引先做治理刷新命令

- 背景：全站搜索已经扩展到活动、AI 社区、影像库，下一步需要让治理助手定期生成结构化摘要、关键词和向量兼容索引。
- 触发：如果直接在 Web 进程里启动后台定时写库，会引入多实例重复执行、部署时机、失败重试和线上数据库写入节奏等问题；但如果只停留在 SQL 召回，又无法验证真实库里的智能检索方向。
- 过程：决定先新增 `resource_search_index` 作为统一索引表，并提供 `refresh-resource-search-index.js` 命令。命令负责执行迁移、读取公开已审核资源、生成摘要/关键词/分面/本地轻量向量、记录 `global_search_index` 刷新运行。部署层或 cron 后续可以调用这个命令，而不是让普通请求进程默认定时写库。
- 结论：`globalSearchService` 保留原 SQL 召回，同时合并 `resource_search_index` 补充召回；当前 `vector_json` 是可替换的本地词元哈希向量，不宣称等同真实 embedding/RAG。
- 复用：后续接入模型 embedding、外部向量库或真正的治理定时任务时，优先替换 `resourceSearchIndexService` 的向量生成/检索实现和部署调度，不改变 `/api/search` 的前端结果协议。
- 位置：`server/src/services/resourceSearchIndexService.js`、`server/scripts/refresh-resource-search-index.js`、`server/src/services/globalSearchService.js`。
