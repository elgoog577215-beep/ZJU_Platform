## 目标

全方位提升活动 AI 推荐助手，让它从“能推荐活动”升级为“可解释、可评测、强制经过 AI 协同分析的推荐 agent”。

本轮重点不是重做聊天 UI，而是提升推荐质量链路：

- 大模型必须参与活动推荐主链路的意图理解和语义排序。
- 本地标准库负责快速召回、硬约束校验和安全兜底。
- 活动 AI 画像索引作为模型前置上下文，减少让模型全量扫活动的成本。
- 推荐结果必须说明为什么相关、哪些条件匹配、哪些条件不确定。
- 压力评测必须覆盖日期、学院/组织、校区、收益、澄清、模型失败等关键场景。

## 背景

当前项目已经具备统一 AI runtime、活动画像索引、活动推荐接口、反馈记录和后台 agent registry。问题不在于缺少入口，而在于活动推荐助手仍有几处成熟度不足：

- 部分生产路径会优先使用本地规则意图解析，模型主要参与后续排序，用户会感觉“AI 能力不够强”。
- 候选池召回依赖有限关键词和固定权重，对“某一天、某个学院、某个收益”的组合约束不够稳定。
- 模型输出虽然会校验 ID，但缺少更明确的“为什么这个排第一、为什么其他排后面”的用户可读解释。
- 现有评测偏基础 smoke test，缺少高压场景来持续证明推荐质量。

## 范围

本次包含：

- 改造活动推荐主链路，使生产请求默认进入统一 AI 协同路径。
- 强化意图解析输出契约，补充日期、组织、硬约束和澄清判断。
- 强化候选池召回：先本地快速召回，再用活动 AI profile 和硬约束评分收窄给大模型。
- 强化模型重排提示词与输出校验，要求返回排序依据、不确定项、匹配信号和信心值。
- 补充评测脚本，覆盖多轮和失败场景。
- 前端轻量展示 AI 分析状态、排序依据和不确定项。

本次不包含：

- 不新增数据库迁移。
- 不重做活动页面整体视觉系统。
- 不把所有历史活动全量塞给模型。
- 不把 API key 写入前端或代码仓库。

## 风险

- 外部模型不可用时，推荐链路必须降级到活动画像索引和本地排序，并明确提示。
- 模型输出可能包含非法 ID、空 JSON 或过长解释，必须继续校验和裁剪。
- 强制模型参与会增加延迟，因此必须控制候选池大小和 token 预算。
- 日期自然语言解析不能假装全能，本轮只做稳定常见表达和评测覆盖。

## 验收标准

- 活动推荐主链路的正常推荐结果包含 `event_recommendation_intent` 和 `event_recommendation_rerank` 等 AI 任务状态。
- 用户提出“某一天、某学院/组织、某校区、某收益”的组合需求时，排序优先满足硬约束最多的活动。
- 模型失败时仍返回可用 fallback 推荐，并明确 warning 和 `fallbackUsed`。
- 前端能展示 AI 排序依据和不确定项，不只显示活动卡片。
- 新增或更新的评测脚本可以证明：模型参与、约束排序、澄清、fallback、非法 ID 防护均有效。
