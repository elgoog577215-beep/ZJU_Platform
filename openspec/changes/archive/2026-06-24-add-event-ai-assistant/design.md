## Context

活动页当前已经具备手动搜索、标签筛选、属性筛选、生命周期筛选和排序能力，核心入口位于 [src/components/Events.jsx](/Users/littlefairy/projects/sqtp1/ZJU_Platform/src/components/Events.jsx)，事件数据通过通用资源接口从 [server/src/controllers/resourceController.js](/Users/littlefairy/projects/sqtp1/ZJU_Platform/server/src/controllers/resourceController.js) 获取。现有系统更像“用户自己翻译需求并操作筛选器”，而不是“系统帮助用户把模糊需求变成候选活动”。

项目已经有服务端调用 LLM 的先例，配置模式集中在 [server/src/utils/wechat.js](/Users/littlefairy/projects/sqtp1/ZJU_Platform/server/src/utils/wechat.js)。但活动推荐与公众号解析的风险模型不同：推荐面对的是用户查询和活动内容的双重不可信输入，因此不能简单复用“把全文扔给模型”的策略。

当前探索得到的产品边界如下：
- MVP 是轻量推荐助手，不是开放式聊天 agent。
- 默认只面向公开、未删除、未开始的活动推荐。
- 如果当前没有未开始活动，先明确告知“暂无未开始活动”。
- 只有用户继续追问时，才放宽到进行中或历史活动。
- 允许最多一轮澄清，避免产品形态滑向多轮聊天。

## Goals / Non-Goals

**Goals:**
- 让用户可以用自然语言描述活动需求，并在活动页快速得到 3-5 个相关活动。
- 在必要时最多追问一轮，以补充关键偏好或约束。
- 将 LLM 的决策范围限制在服务端构建的候选池内，且候选数据仅包含公开摘要字段。
- 复用现有活动详情交互，让推荐结果可以直接打开详情继续判断。
- 在无结果、模型异常或候选校验失败时，优先安全降级，而不是返回不可信结果。

**Non-Goals:**
- 不构建一个可长期多轮对话的活动聊天机器人。
- 不让模型直接读取活动原始 `content` HTML、未审核活动或被删除活动。
- 不引入向量数据库、Embedding 检索或个性化用户画像推荐。
- 不让助手替用户执行报名、收藏或其他副作用操作。

## Decisions

### 1. 候选池由服务端构建，并以“公开 upcoming”作为默认边界

默认候选池固定为：
- `status = approved`
- `deleted_at IS NULL`
- 开始时间晚于或等于当前时间；但当活动只有日期、没有具体时间时，只要日期等于当前本地日期，仍视为可推荐候选

当用户在“暂无未开始活动”的反馈后继续追问时，服务端才允许按顺序放宽到“进行中”活动，再放宽到“历史已结束”活动。

选择这个方案，是为了把可见性边界和推荐边界放在同一个可信层。前端不负责拼接候选活动，也不向模型声明“你可以看哪些内容”。

备选方案：
- 让前端把当前列表数据直接传给模型：实现快，但容易继承页面缓存、状态过滤和可见性错误。
- 让模型直接查询数据库或自由搜索活动：权限过大，不符合 MVP 的安全目标。

### 2. 模型只看活动摘要，不看原始详情正文

传给模型的每个候选活动只包含受控摘要字段：
- `id`
- `title`
- `description`
- `date`
- `end_date`
- `location`
- `tags`
- `organizer`
- `target_audience`
- `score`
- `volunteer_time`

活动原始 `content`、HTML、外链正文和其他未清洗富文本不进入模型上下文。这样即便活动内容来自用户上传或公众号抓取，也不会直接成为模型指令面的一部分。

备选方案：
- 直接传全文 `content`：语义信息更丰富，但 prompt injection 风险更高，输出也更难校验。
- 完全不使用活动摘要，只让模型做标签映射：安全，但推荐质量过于依赖已有标签体系，无法覆盖自然语言细节。

### 3. 接口采用严格结构化输出，而不是开放式聊天文本

推荐接口返回三种结构化结果之一：
- `clarify`: 需要补充信息时返回一个澄清问题
- `recommend`: 返回候选池内的 3-5 个活动 ID 及对应短理由
- `empty`: 当前作用域下没有可推荐活动

服务端必须校验：
- 返回类型是否合法
- 推荐活动 ID 是否存在于当前候选池
- 推荐数量是否在上限内
- 澄清轮次是否超过 1 轮

这样做是为了把“模型理解意图”和“系统相信结果”分开。模型可以给建议，但系统只接受符合协议的建议。

当当前候选池不足 3 个活动时，系统返回全部可用候选，而不是为了凑数扩大作用域或编造结果。

备选方案：
- 让模型直接返回富文本答案：前端渲染更轻松，但难以约束活动 ID、追问轮次和空结果策略。

### 4. 对话形态限制为“初始请求 + 最多一轮澄清”

前端保留一个很轻的本地会话状态，只记录：
- 初始用户请求
- 助手是否要求澄清
- 用户的澄清回答
- 当前是否允许放宽搜索范围

服务端每次根据请求重建候选池并调用模型，不依赖长期会话存储。这样可以避免引入新的会话表、定时清理逻辑或复杂状态同步。

备选方案：
- 服务端持久化推荐会话：更严谨，但对 MVP 偏重。
- 完全无澄清：实现最简单，但对模糊需求的容错较差。

### 5. 推荐结果复用现有活动详情交互，而不是另起一套详情页

推荐结果只需要展示少量活动卡片或结果列表，并提供简短推荐理由。用户点击结果后，沿用活动页已有的详情打开方式，这样可以避免重复建设详情数据流，也能保证推荐结果与手动浏览体验一致。

备选方案：
- 助手消息内直接展开完整详情：实现上容易重复渲染逻辑，也会让助手区域过重。

## Risks / Trade-offs

- [候选池时间边界判定不一致] → 统一由服务端定义 upcoming / ongoing / past，避免前端和模型各自判断。
- [模型返回不存在的活动或编造理由] → 服务端按候选池 ID 白名单校验；校验失败时降级为空结果或通用失败提示。
- [活动摘要信息不够，导致推荐偏粗] → MVP 接受较保守的推荐质量，后续若确有需要，再引入服务端生成的安全摘要字段，而不是直接开放全文。
- [无 upcoming 活动时体验容易“空”] → 明确返回空状态，并允许用户继续追问后放宽范围，而不是默认偷偷扩大作用域。
- [现有页面查询存在 `status=all` 的历史用法] → 助手接口独立执行可见性过滤，不复用页面当前已加载列表作为候选池来源。
- [LLM 输出格式不稳定] → 使用严格的结构化协议和服务端兜底文案，不让格式错误直接泄漏到 UI。

## Migration Plan

该变更不涉及数据库 schema 迁移。

建议按以下顺序落地：
1. 先新增服务端推荐接口和结构化输出校验。
2. 再在活动页接入助手 UI，并接通现有详情弹层。
3. 以本地开发或测试数据验证“有 upcoming / 无 upcoming / 继续追问放宽范围”三条主路径。

回滚策略应保持简单：
- 若助手接口或 LLM 不稳定，可直接隐藏活动页 AI 助手入口，不影响原有手动筛选流程。
