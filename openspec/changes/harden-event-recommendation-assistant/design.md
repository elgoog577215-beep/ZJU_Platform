## Context

活动推荐助手经历过多轮建设：最初是活动页自然语言推荐入口，之后接入统一 AI runtime、活动智能标准库、活动 AI 画像、用户偏好、反馈记录、运行遥测和后台 Agent registry。当前能力已经超过“Beta 搜索”，但实现形态仍有明显问题：

- `server/src/utils/eventAssistant.js` 承担过多职责，推荐流程、模型调用、排序、记忆、反馈和响应组装混在一个文件中。
- 活动 AI 画像、反馈证据和统一 runtime 已存在，但推荐主链路没有把它们抽象成稳定服务边界。
- 用户画像保存和反馈记录已经具备基础表，但缺少“显式偏好、行为信号、对话记忆、负反馈”的统一权重模型。
- 前端展示了大量 AI 状态和解释信息，但主流程还不够像一个“可纠错的助手”。
- 已有脚本覆盖 smoke、stress 和 live eval，但缺少固定 golden case 来防止推荐质量回退。

本设计目标是做质量治理，不是重新发明现有 AI runtime 或重做整个活动页面。

## Goals / Non-Goals

**Goals:**

- 把活动推荐助手拆成可理解、可测试、可替换的服务层。
- 让确定性推荐成为质量底座，模型只在受控候选池内增强语义理解、解释和有限重排。
- 让用户画像、反馈证据和活动画像真实影响排序，并在响应中可解释。
- 让前端主流程更清晰：用户知道助手理解了什么、为什么推荐、哪里不确定、如何纠正。
- 建立 golden eval、fallback eval 和 UI smoke 验收，保证后续迭代不靠主观感觉。

**Non-Goals:**

- 不重做活动 CRUD、报名、收藏、详情页或活动筛选系统。
- 不引入向量数据库、后台任务队列或新的模型供应商。
- 不允许模型直接查询数据库或生成数据库外活动。
- 不自动报名、自动收藏或自动订阅提醒。
- 不在本轮强制新增数据库迁移；除非实现时证明现有表无法表达必要证据。

## Decisions

### 1. 拆分推荐助手服务边界

将 `eventAssistant.js` 中的核心职责拆成服务模块：

- `EventIntentService`：本地规则 + 模型意图理解，输出规范化需求。
- `EventRetrievalService`：加载活动池、生命周期分组、基础过滤和历史 fallback 策略。
- `EventRankingService`：确定性评分、硬约束保护、画像加权和负反馈惩罚。
- `EventProfileService`：聚合用户显式偏好、收藏、报名、浏览、反馈和对话记忆。
- `EventExplanationService`：基于已排序候选生成摘要、推荐理由、不确定项和提示语。
- `EventAssistantTelemetryService`：记录运行摘要、fallback、评分信号和评测指标。
- `EventAssistantResponseBuilder`：统一响应契约，避免前端适配多个形态。

保留 `eventAssistant.js` 作为兼容门面，先只负责调用新服务和导出既有函数。这样可以保持 `/api/events/assistant` 和测试注入能力稳定。

替代方案是继续在原文件内局部整理。这个方案改动小，但无法解决长期维护问题，也会让后续画像、反馈和评测继续耦合。

### 2. 确定性排序先于模型解释

推荐质量底座采用本地可测试评分：

```text
finalScore =
  lifecycleScore
  + hardConstraintScore
  + topicCategoryScore
  + audienceCampusBenefitScore
  + userProfileScore
  + behaviorSignalScore
  + popularityFreshnessScore
  + diversityBoost
  - explicitMismatchPenalty
  - negativeFeedbackPenalty
  - historicalPenalty
```

硬约束来自用户显式表达，例如“本周”“紫金港”“计算机学院”“有综测”“线下”。硬约束不是绝对过滤，因为活动数据可能缺字段，但必须强烈影响排序，并进入 `matchSignals` 和 `reasoningTrace.rankingBasis`。

替代方案是让模型直接重排全部候选。这个方案演示效果可能更强，但不可测、延迟高，也容易忽略数据库字段和硬约束。

### 3. 用户画像分四类信号

画像服务输出统一 `profileSignals`：

- 显式偏好：`user_event_preferences` 中的学院、年级、校区、兴趣、类型和收益偏好。
- 行为信号：收藏、报名、浏览、打开详情等近期行为。
- 对话记忆：`assistant_memory` 中用户明确要求记住的偏好。
- 负反馈：`event_recommendation_feedback` 中的 down、原因和对应推荐信号。

画像服务只向模型提供摘要，不传原始行为列表。排序层使用权重，解释层使用短文本信号。负反馈优先降低类似推荐，不直接永久屏蔽某类活动。

替代方案是只用显式偏好。它更简单，但无法形成“越用越准”的助手。

### 4. 模型职责受限

模型可以做三件事：

- 将自然语言需求补全成结构化意图。
- 在后端候选池内做有限语义重排。
- 生成用户可读解释、澄清问题和不确定项。

模型不能：

- 推荐候选池外活动 ID。
- 覆盖后端硬约束保护结果。
- 编造活动状态、报名链接、奖励、地点或主办方。
- 直接写入用户长期记忆。

后端继续校验模型 JSON、活动 ID、数量、字段长度、置信度范围和非法输出。模型失败时返回本地 fallback 推荐，并明确 `modelStatus.fallbackUsed` 和 warning。

### 5. 前端从“信息面板”收束为“推荐工作流”

前端保留现有入口，但调整信息层级：

1. 顶部输入区：突出“说出你要找什么”，快捷提示作为辅助。
2. 理解区：展示助手理解到的主题、时间、校区、收益、对象和画像参考。
3. 推荐区：每张卡片只保留核心理由、关键匹配信号、风险标签、时间地点和打开详情。
4. 纠错区：反馈从简单赞踩扩展为原因选择，例如“不相关”“时间不合适”“地点不合适”“收益不符合”“已经参加”。
5. 状态区：模型参与、fallback、历史推荐等状态低调但明确，不抢主流程。

移动端继续使用全屏助手，但保持同一响应契约，避免移动端和桌面端分叉。

### 6. 评测先覆盖质量回退风险

建立三类验证：

- 单元/脚本验证：意图解析、生命周期、排序、非法模型输出、fallback、画像影响。
- golden eval：固定 20 到 50 条校园活动查询，断言前几名应满足哪些显式约束。
- UI smoke：桌面和移动端能完成输入、推荐、打开详情、反馈、空状态和 fallback 展示。

现有脚本可以继续复用，但本轮应把评测结论变成稳定验收，而不是一次性检查。

## Risks / Trade-offs

- [拆分文件带来短期改动面] → 先保留兼容门面，按服务逐步迁移，每一步跑现有脚本。
- [确定性排序过强导致结果不够“惊喜”] → 用 diversityBoost 和模型解释补足探索性，但显式需求优先。
- [行为画像可能误判兴趣] → 使用时间衰减和低权重，显式偏好与当前查询始终高于历史行为。
- [负反馈原因不足] → 前端先增加有限原因选项，后端兼容旧的 up/down。
- [模型失败影响体验] → fallback 推荐必须可用，并清楚说明模型解释暂不可用。
- [中文编码影响 prompt 与 UI] → 执行时必须检查关键中文文件的实际编码和浏览器展示，不只看终端输出。

## Migration Plan

1. 新增服务模块并保持 `runEventAssistantTurn` 对外签名不变。
2. 把现有逻辑迁移到服务层，每迁移一层就跑现有 assistant 验证脚本。
3. 加入确定性评分解释和 profile signal 输出，保持响应向后兼容。
4. 增强反馈原因和前端展示，旧反馈数据继续可读。
5. 补 golden eval、fallback eval、UI smoke。
6. 验证通过后再考虑归档旧 OpenSpec 变更，避免主线 specs 与已完成 change 长期分裂。

Rollback：

- 若服务拆分导致生产接口异常，可以让 `eventAssistant.js` 回退调用旧实现。
- 若新排序策略质量不稳定，可以保留新响应字段但切回旧评分权重。
- 若前端反馈原因影响用户操作，可以隐藏原因选择，继续只提交 up/down。

## Open Questions

- 是否需要把“打开详情”作为隐式正反馈写入 `event_recommendation_feedback`，还是只从运行日志与收藏报名间接推断？
- 是否需要新增 `feedback_reason` 的枚举约束，还是继续用自由文本 `reason` 兼容前端？
- 活动画像刷新是否仍采用请求时补齐，还是后续单独引入后台刷新任务？
