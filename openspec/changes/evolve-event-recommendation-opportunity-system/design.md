## Context

活动推荐助手当前已经具备统一 AI runtime、活动画像、硬约束保护、负反馈记录、规则补齐和 golden eval。它的问题不再是“能不能返回推荐”，而是目标层级还停留在推荐功能：响应告诉用户“推荐了什么”，但没有稳定表达“为什么这个机会值得你参加、哪里不确定、为什么它优于其他候选、你的反馈如何改变下一轮”。

本轮设计把活动推荐助手推进为校园机会匹配系统的第一阶段。它不是一次性完成全部智能入口，而是在现有 API 兼容的前提下，新增可执行、可观测、可评测的机会匹配契约。

## Goals / Non-Goals

**Goals:**

- 固化北极星目标和五阶段目标，让后续迭代有稳定验收方向。
- 在响应中提供结构化 `opportunityMatch`，覆盖匹配项、缺失项、不确定项、决策提示、反馈学习状态。
- 让负反馈原因影响排序和解释，尤其是时间、地点、收益、相关性、已参加等明确原因。
- 增加轻量对比解释，帮助用户理解为什么推荐第一项而不是其他候选。
- 扩展运行摘要和 golden eval，使目标不只存在于文档，而能被脚本持续检查。

**Non-Goals:**

- 本轮不新增数据库迁移，不改变公开 API 路径。
- 本轮不做复杂协同过滤、向量数据库或长期训练。
- 本轮不把活动助手扩展到首页、通知或运营后台的完整分发系统，只预留可观测指标和响应契约。
- 本轮不承诺真实线上统计达到学期级北极星目标，只建立第一阶段可验证闭环。

## Decisions

### 1. 用追加字段升级 API，而不是替换响应结构

保留现有 `recommendations`、`reasoningTrace`、`modelStatus`，新增 `opportunityMatch` 和必要诊断字段。这样前端可渐进展示，旧调用方不会被破坏。

替代方案是重写响应结构为“机会卡片”。这会更干净，但会破坏现有面板、e2e 和外部兼容，不适合当前阶段。

### 2. 负反馈原因先走规则权重，不新增表结构

前端已经把负反馈原因拼入 `reason`。本轮后端解析该文本，归纳为 `not_relevant`、`time_mismatch`、`location_mismatch`、`benefit_mismatch`、`already_joined` 等信号，并在评分中对相同活动、相同类别、相同缺口进行惩罚。

替代方案是新增枚举字段 `feedback_reason`。这更规范，但需要迁移和兼容旧数据；当前先用文本解析形成行为闭环。

### 3. 决策解释由后端确定性生成，模型只提供候选重排

“为什么 A 优于 B”应基于硬约束分、匹配信号、缺失项和活动事实生成，避免模型编造比较理由。模型仍参与语义排序，但决策解释不依赖模型自由发挥。

### 4. 目标观测写入运行摘要

`ai_assistant_runs.summary_json` 已存在。本轮追加机会匹配阶段、硬约束均值、缺失项、反馈学习是否使用、规则补齐数量等字段，不新增表。

## Risks / Trade-offs

- [Risk] 负反馈原因来自自由文本，解析不可能完全准确。→ 只识别有限枚举中文短语，识别不到时退回普通负反馈，不影响主流程。
- [Risk] 结构化解释变多后前端信息过载。→ 前端只展示最关键的匹配/缺失/决策提示，完整诊断留在响应和评测。
- [Risk] 决策解释可能显得机械。→ 优先保证真实、可验证；文案可以后续再打磨。
- [Risk] 北极星目标太远大，单轮无法完成。→ 本轮明确只完成第一阶段闭环，并把后续目标写入 spec 与任务边界。

## Migration Plan

1. 新增响应字段和运行摘要字段，保持旧字段不变。
2. 前端读取新字段时做存在性判断，旧响应仍可展示。
3. golden eval 增加断言，验证中文真实问法、负反馈原因、决策解释和运行摘要。
4. 如发现字段缺失或旧数据库没有运行表，沿用现有 try/catch 降级策略。

## Open Questions

- 后续是否需要把 `feedback_reason` 独立成数据库字段，以支持更可靠的运营分析？
- 首页、通知和运营后台什么时候接入机会匹配摘要？
- 是否需要把“打开详情但未报名”也作为弱反馈信号？
