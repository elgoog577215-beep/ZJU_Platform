## Context

当前主题系统由三层混合构成：

1. `SettingsContext` 负责保存 `uiMode`，并通过 `data-theme` 挂到 `documentElement`。
2. `src/index.css` 提供一批 `--theme-*` 变量，但白天模式同时又通过全局类名覆写去反转深色类。
3. 大量组件内部继续使用 `isDayMode ? ... : ...` 手写颜色、边框、遮罩和阴影。

这导致白天模式既不是独立主题，也不是稳定的 token 消费体系，而是“深色设计 + 补丁覆盖”。从实测页面看，首页首屏、社区页、视频页和后台容器都存在同类问题：画面发白、层级失焦、卡片与背景边界不清、夜间压图逻辑残留。

用户已经确认本次改造范围覆盖前台与管理后台，目标是“视觉整改 + 主题体系重构”，并且允许白天模式与黑夜模式在关键区块上明显分化。额外审美约束是：白天模式必须具备更高的 UI 审美标准，呈现出艺术家的精致感，而不是标准化企业后台或简单浅色换肤。

## Goals / Non-Goals

**Goals**

- 建立一个可被前台与后台共同消费的语义化主题层。
- 让白天模式在首页、导航、卡片、弹层、列表和后台高频界面中形成独立、稳定、精致的表达。
- 减少继续扩散的全局反向类名覆写与组件内硬编码颜色。
- 为白天模式建立明确的视觉验收与回归标准，包括移动端和桌面端。

**Non-Goals**

- 不在本次 spec 中发起一次全量组件架构重构。
- 不引入新的状态管理库或设计系统框架。
- 不要求所有遗留组件一步完成彻底清洗；本次优先收敛高频路径和共享基础层。
- 不改变主题切换入口、路由结构或后台信息架构。

## Decisions

### 1. 建立共享语义 token 层，停止继续扩大反向覆写

保留现有 `data-theme` 机制，但将白天/黑夜模式的基础变量从“少量颜色变量 + 大量类名覆写”升级为“语义 token + 兼容层”。

新增或明确的 token 分组至少包括：

- `--bg-base`, `--bg-subtle`, `--bg-elevated`
- `--surface-default`, `--surface-strong`, `--surface-hover`
- `--border-default`, `--border-strong`
- `--text-primary`, `--text-secondary`, `--text-muted`, `--text-inverse`
- `--overlay-soft`, `--overlay-strong`
- `--shadow-sm`, `--shadow-md`, `--shadow-lg`
- `--accent-default`, `--accent-soft`, `--accent-contrast`

前台与后台的高频组件优先消费上述 token。现有 `.text-white`、`.bg-black/20`、`.border-white/10` 这类白天模式反向覆写不再扩大，只作为过渡兼容层；后续新改组件不得继续依赖这类覆写获得正确外观。

选择该方案是因为它能在不推翻现有结构的前提下，把主题演进路径从“局部修色”转成“稳定消费语义层”。

备选方案：

- 继续沿用 `isDayMode` 分支逐页修补：短期快，但不可维护。
- 直接上完整设计系统改造：理论更整洁，但范围过大、回归面过宽。

### 2. 白天模式允许与黑夜模式在关键区块上明显分化

白天模式不追求“与黑夜模式同构，仅换配色”，而是在关键区块上允许结构和层级表达不同，只要求交互和信息架构保持一致。

关键分化区块包括：

- 首页首屏
- 顶部导航
- 内容卡片与媒体卡
- 列表筛选区与搜索面板
- 详情弹层 / 抽屉 / 模态框
- 后台卡片、表格、表单和筛选区

例如：首页首屏在黑夜模式下可以保留沉浸式背景与发光感，但白天模式必须转向高可读、低雾化、强焦点的排版与图像关系；后台白天模式则必须以功能清晰、边界稳定为主，不复用前台装饰性光晕。

这样做是为了满足用户对审美的明确要求：白天模式要有自己的艺术语言，而不是夜间布局的浅色复制品。

### 3. 背景系统与首屏特效按模式拆分约束

`BackgroundSystem` 和 `Hero` 的视觉特效不再共享同一组强度逻辑。白天模式必须采用更克制的背景策略：

- Bloom 默认关闭或显著降低
- Vignette 默认关闭或极轻
- 顶部发光球与大面积彩色光晕透明度减半
- 首页首屏禁止出现深色压边和近白标题压在浅背景上的组合
- 白天模式的背景图、渐变和遮罩必须优先服务于阅读焦点，而不是沉浸感

该决策直接解决当前首页首屏和内容页“泛白、失焦、夜间残留”的主要来源。

### 4. 建立高频组件的主题消费规则，而不是一次性全量清洗

本次实现按组件优先级推进：

第一层，基础和全局：

- `src/index.css`
- `BackgroundSystem`
- `Hero`
- `Navbar`
- 公共按钮、输入框、面板、弹层基类

第二层，前台高频页面：

- 首页与内容列表页
- 搜索面板
- 上传弹层
- 公共详情弹层
- 个人页

第三层，后台高频页面：

- Dashboard 容器
- 卡片、表格、分页、筛选区、弹窗、表单

每个组件允许保留少量 `isDayMode` 分支处理结构差异，但色彩、阴影、边框、遮罩必须尽量改为 token 驱动。这样可以在可控范围内完成全站改造，而不是陷入一次性清洗所有历史分支。

### 5. 将“艺术家的精致感”落成可验收设计标准

审美要求不能只写成主观描述，本次设计将其转为以下约束：

- 白天模式优先采用更克制的色彩数量，避免同时存在多块高饱和大光晕。
- 大面积浅背景中的焦点元素必须通过版式、对比和容器关系建立，而不是靠粗暴发光。
- 卡片、导航和弹层的阴影要轻而明确，不能出现一层雾面把边界全部磨平。
- 图像卡片在白天模式下不得继续复用夜间的重黑罩 + 白字逻辑，除非该卡片本身需要媒体封面式表达。
- 后台白天模式必须体现“精致但克制”的专业感，不能退化成默认 Tailwind 面板堆叠。

## Risks / Trade-offs

- 旧组件与新 token 并存期间，局部页面可能同时受 token 和兼容覆写影响，需要逐页检查。
- 白天模式允许结构分化后，前后台测试面会增加，需要通过截图回归压住风险。
- 如果白天模式追求艺术感过头，容易牺牲后台效率，因此后台设计必须优先清晰度和可扫描性。
- 若只改前台、不同时推进后台，主题系统仍会分裂，因此本次必须保留后台范围。

## Migration Plan

该变更不涉及数据库迁移。

建议按以下顺序落地：

1. 定义并替换基础 token，收口全局覆写范围。
2. 完成 `BackgroundSystem`、`Hero`、`Navbar` 与公共面板/表单/弹层基类。
3. 完成前台高频页面白天模式整改。
4. 完成后台高频容器和表单/表格白天模式整改。
5. 补 Playwright 回归基线，并在桌面端与移动端验证白天/黑夜双主题。

## Open Questions

无。当前范围、改造原则和审美标准已由用户明确确认，可直接进入任务拆解和实现。
