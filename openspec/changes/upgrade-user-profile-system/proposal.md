## Why

当前用户系统已经支持登录、昵称、关注关系和个人主页内容聚合，但用户身份表达仍然不完整：头像上传入口不够正式，比赛成果虽然已有投稿与展示页面，却没有稳定沉淀为“用户个人成果”。这会让用户主页更像内容列表，而不是可持续积累的作品履历。

随着比赛成果模块、社区内容和个人主页逐渐成为平台核心使用路径，需要把“用户是谁、代表什么身份、发布了什么、取得了什么成果”串成一条清晰链路。尤其是平台自办活动中，成果可能由管理员批量发布，发布者不等于获奖者或所属团队，因此成果归属需要从“谁上传”升级为“谁被成果文本指向，并由当事人确认绑定”。

## What Changes

- 完善登录用户头像上传能力，支持选择、预览、上传、保存和替换头像。
- 统一头像展示来源，继续以 `users.avatar` 作为用户头像的唯一持久化字段。
- 将比赛成果正式纳入用户主页内容聚合，让用户主页展示已确认归属到该用户、社团或团队身份的 `competition_works`。
- 将 `uploader_id` 定义为发布者/提交者字段，仅用于审计、投稿管理和通知，不作为成果最终归属依据。
- 引入可选身份认证/身份声明能力，不只支持个人真实姓名，也支持社团、团队等组织身份。
- 支持基于成果中的获奖者姓名、团队名、社团名等文本信息生成候选绑定，但绑定是否生效需要当事人确认通过。
- 为成果绑定提供候选、确认、拒绝、撤销等状态，避免重名、团队重名或误匹配造成错误展示。
- 为用户维度提供已确认成果查询能力，区分访客、本人、管理员的可见范围。
- 在个人主页中为比赛成果提供独立内容类型和更适合成果展示的卡片信息。
- 保留现有登录、JWT 鉴权、管理员审核和比赛成果投稿主流程，不在本变更中重做认证系统或审核系统。
- 不把文本匹配结果直接视为最终归属，所有自动匹配都必须经过当事人确认或管理员辅助确认。
- 不强制所有用户进行身份认证；未认证用户仍可正常使用平台，只是无法自动确认某些外部成果归属。

## Capabilities

### New Capabilities

- `user-avatar-upload`: 登录用户可以上传头像，后端校验文件类型和大小，保存头像 URL，并在个人资料、作者信息和当前登录用户状态中体现。
- `user-identity-claims`: 用户可以声明和认证个人、社团或团队身份，为后续成果归属确认提供可信身份来源。
- `user-competition-outcomes`: 用户可以通过已确认身份绑定拥有比赛成果；系统按绑定关系查询、展示和控制可见性，使比赛成果成为个人主页的一类正式资产。

### Modified Capabilities

- `user-profile-content-aggregation`: 个人主页内容聚合范围扩展到已确认绑定的比赛成果，资源列表中需要包含可见的 `competition_work` 项。
- `community-author-identity`: 用户头像和显示名继续作为作者身份的统一展示来源，优先使用昵称，缺省时回退到用户名。

## Impact

- Frontend:
  - `src/components/PublicProfile.jsx`：个人主页头像设置区、身份声明入口、内容类型筛选、比赛成果卡片、绑定状态展示与跳转。
  - `src/services/api.js`：头像上传、身份声明、成果绑定确认、用户资源/成果查询调用。
  - 比赛成果相关页面：需要展示原始获奖者/团队文本，以及已确认绑定的用户、社团或团队身份。

- Backend:
  - `server/src/middleware/upload.js`：头像专用上传限制与存储目录。
  - `server/src/controllers/userController.js`：头像保存、身份声明、绑定确认、用户成果查询、个人资源聚合。
  - `server/src/routes/api.js`：用户头像上传、身份声明、绑定确认与用户成果查询路由。
  - `server/src/config/runMigrations.js`：为身份声明和成果绑定补充非破坏性表结构与索引。

- Database:
  - 复用 `users.avatar`。
  - 复用 `competition_works.uploader_id`、`author`、`status`、`public_consent`、`deleted_at`。
  - 新增身份声明/认证相关数据结构，用于保存个人姓名、社团名、团队名等可选身份。
  - 新增比赛成果与身份/用户之间的绑定关系数据结构，用于保存候选、已确认、已拒绝、已撤销等状态。
  - 仅允许新增非破坏性表结构和索引，不进行破坏性迁移。

- Verification:
  - 验证头像上传成功后 `/auth/me` 和个人主页头像同步更新。
  - 验证成果文本命中个人姓名、社团名或团队名时，只生成候选绑定，不直接公开归属。
  - 验证当事人确认绑定后，成果才进入对应用户主页。
  - 验证当事人拒绝或撤销绑定后，成果不再作为该用户成果展示。
  - 验证访客只能看到已审核、允许公开且已确认绑定的比赛成果。
  - 验证本人和管理员可以看到相关候选/已确认/已拒绝绑定状态。
  - 验证普通内容、社区内容和比赛成果在个人主页中不会互相破坏排序、筛选和跳转。
