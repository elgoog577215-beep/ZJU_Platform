## Context

用户资料当前由 `users` 表、`AuthContext`、`PublicProfile` 和 `/api/users/:id/*` 接口驱动。平台已有 `users.avatar`、`competition_works.uploader_id`、比赛成果审核状态和个人主页资源聚合，但这些能力还没有形成完整的“用户身份 + 成果归属”系统。

关键约束是：比赛成果的发布者不一定是成果拥有者。平台自办活动中，管理员可能批量录入获奖作品，此时 `uploader_id` 指向管理员，只能说明谁创建了记录，不能说明成果属于谁。成果归属应由成果文本中的获奖者姓名、团队名、社团名等身份线索，与用户主动声明的身份建立候选关系，再由当事人确认后生效。

## Goals / Non-Goals

**Goals:**

- 登录用户可上传头像，并同步刷新当前用户资料和个人主页展示。
- 用户可声明个人、社团、团队等可选身份，用于后续成果归属确认。
- 系统可基于比赛成果中的获奖者姓名、团队名、社团名等文本，生成候选绑定。
- 候选绑定必须由当事人确认，或由管理员辅助确认后，才作为个人主页成果展示。
- 个人主页展示已确认绑定的比赛成果，同时保留原始成果署名文本。
- 复用现有登录、JWT、SQLite、上传、安全过滤和管理员审核体系。

**Non-Goals:**

- 不强制所有用户认证真实姓名或组织身份。
- 不把 `uploader_id` 当作成果归属依据。
- 不让自动文本匹配直接公开绑定成果。
- 不做复杂证件实名、人脸识别、学校统一身份认证或 OAuth。
- 不清洗所有历史成果数据；历史数据只在有可匹配文本和用户主动声明身份后进入候选绑定流程。
- 不在本次实现头像裁剪、对象存储、复杂团队成员权限或完整组织后台。

## Decisions

### 1. 区分“发布者”和“归属者”

`competition_works.uploader_id` 继续保留，但语义限定为发布者/提交者，用于审计、通知、投稿管理和管理员追溯。成果是否出现在某个用户主页，不再由 `uploader_id` 决定，而由成果绑定关系决定。

备选方案：继续使用 `uploader_id`。放弃原因是管理员批量发布会把成果错误归到管理员名下，无法表达团队、社团或多人获奖。

### 2. 新增身份声明模型

新增用户身份声明数据结构，用于保存用户主动声明的身份。建议字段语义包括：

- `id`
- `user_id`
- `type`: `person`、`team`、`club`
- `display_name`
- `normalized_name`
- `status`: `pending`、`verified`、`rejected`
- `created_at`、`updated_at`

第一阶段可以允许用户自助创建身份声明，并将状态设为 `pending` 或轻量 `verified`，具体认证严格程度由设计后的任务再细化。重点是先建立身份实体，而不是把真实姓名直接塞进 `users` 表。

备选方案：在 `users` 表新增 `real_name`、`team_name`、`club_name`。放弃原因是一个用户可能代表多个团队/社团身份，单字段难以扩展，也难以表达状态和审核记录。

### 3. 新增成果绑定关系模型

新增比赛成果与身份/用户之间的绑定关系。建议字段语义包括：

- `id`
- `work_id`
- `claim_id`
- `user_id`
- `matched_text`
- `match_source`: `auto`、`manual_user`、`manual_admin`
- `status`: `candidate`、`confirmed`、`rejected`、`revoked`
- `confidence`
- `created_at`、`updated_at`、`confirmed_at`

`claim_id` 表示绑定到哪个身份声明，`user_id` 用于快速查询用户成果。一个作品可以有多个确认绑定，以支持团队成员、多人获奖、社团成果。

备选方案：在 `competition_works` 直接新增 `owner_user_id`。放弃原因是无法表达一个成果对应多人、多团队、多社团，也无法保留候选、拒绝、撤销状态。

### 4. 文本匹配只生成候选

系统可以从 `competition_works.author`、`honor_title`、`summary`、后续可能的成员字段中提取或搜索身份名称。匹配时使用规范化名称，例如去空格、统一大小写、统一常见全角/半角字符。匹配结果只创建 `candidate`，不能直接作为公开归属。

若同一文本命中多个身份声明，所有命中都保持候选状态，并在用户侧或管理员侧提示需要确认。确认后才进入用户主页成果聚合。

### 5. 当事人确认优先，管理员辅助确认

用户在个人主页或消息中心看到“可能属于你的成果”候选项，可以确认、拒绝或稍后处理。确认后状态变为 `confirmed`，成果可进入个人主页。拒绝后状态变为 `rejected`，不再展示为该用户成果。用户应能撤销已确认绑定，撤销后状态变为 `revoked`。

管理员可以在后台为明显的历史数据辅助创建或确认绑定，但仍应保留操作来源，避免不可追溯的静默归属。

### 6. 个人主页只聚合已确认绑定成果

`GET /api/users/:id/resources` 应包含该用户已确认绑定的 `competition_work`。访客还必须满足成果本身 `status = approved`、`public_consent = 1`、`deleted_at IS NULL`。本人和管理员可以看到更多绑定状态和审核状态，用于管理与排错。

成果卡片应同时展示：

- 原始获奖者/团队文本，例如 `author`
- 已确认绑定身份，例如某用户、团队、社团
- 比赛名称、奖项、排名、封面、摘要、审核/绑定状态

### 7. 头像上传保持独立小闭环

头像继续使用 `users.avatar` 作为唯一持久化来源。上传接口使用 `POST /api/users/me/avatar`，限制 JPG/PNG/WebP、5MB、`uploads/avatars`。上传成功后返回头像 URL 和更新后的用户资料，前端调用 `refreshUser()` 并更新个人页本地状态。

头像上传与身份声明/成果绑定解耦：头像是用户展示资料，身份声明是成果匹配凭据。

## Risks / Trade-offs

- [Risk] 姓名、团队名、社团名重名导致误匹配。 -> Mitigation: 自动匹配只生成候选，必须由当事人或管理员确认后才公开归属。
- [Risk] 用户恶意声明他人姓名或社团身份。 -> Mitigation: 身份声明保留状态字段和审核入口；被争议绑定可撤销，管理员可拒绝或停用可疑声明。
- [Risk] 历史成果文本结构不统一，自动匹配覆盖有限。 -> Mitigation: 第一阶段只做保守精确匹配，并允许管理员手动创建候选或确认绑定。
- [Risk] 个人主页查询变复杂。 -> Mitigation: 为绑定表建立 `user_id/status/work_id` 索引，为身份声明建立 `normalized_name/type/status` 索引。
- [Risk] pending/rejected 成果或候选绑定泄露。 -> Mitigation: 后端统一按 viewer 身份判断，访客只看到 approved + public + confirmed。
- [Risk] 旧头像文件残留。 -> Mitigation: 本次不删除旧头像文件，避免误删；后续可单独做清理任务。

## Migration Plan

- 新增身份声明表和成果绑定表，均为非破坏性迁移。
- 为身份声明名称和成果绑定查询补充索引。
- 保留 `users.avatar`、`competition_works.uploader_id` 和现有比赛成果字段，不删除、不改语义之外的数据。
- 首次上线后不自动公开绑定历史成果，只生成候选绑定或等待用户声明身份后再匹配。
- 回滚时可以停止使用新表和新接口，原有用户、比赛成果、审核和上传流程不受影响。

## Open Questions

- 身份声明第一阶段是否需要管理员审核后才可参与匹配，还是允许用户自助声明后先生成候选、确认展示前再加强审核？
- 团队/社团身份是否需要多个用户共同管理，还是第一阶段只允许声明者代表该身份确认成果？
- 候选绑定通知放在个人主页设置区、通知中心，还是单独的“成果认领”入口？
- 历史成果是否需要一次性扫描生成候选，还是只在用户新增身份声明、管理员编辑成果、成果发布时触发匹配？
