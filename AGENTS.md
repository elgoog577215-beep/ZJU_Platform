# ZJU Platform Agent Guide

本文件约束在本仓库内工作的 AI / agent。目标不是“尽快改完”，而是“改完且可验证，不把回归留给下一个人”。

## Project Context

本仓库服务于拓途浙享、浙江大学 AI 生态团队官网、AI 社区、活动聚合/分类/推荐、黑客松页面/报名页、管理后台、微信解析助手和 AI 活动推荐助手。

核心产品原则：

- 活动分类、学院、组织社团、主办方、面向对象等要进入统一标准库。
- AI 解析优先从标准库选择，不要自由生成关键分类后再二次匹配。
- AI 输出要有结构化结果、置信度、候选项、人工校正入口。
- 历史数据迁移要支持 dry-run、报告、低置信度项人工确认。
- UI 默认要求高：视觉冲击力、呼吸感、精致感、现代感、清晰主次；白天/夜间和移动端/桌面端都要可用。

## Before Editing

1. 先读实际代码和当前 diff，再下结论，禁止靠记忆描述现状。
2. 不要覆盖用户已有改动；遇到无关改动时忽略，遇到相关改动时顺着它工作。
3. 涉及数据、权限、路由、部署、数据库、AI 流程时，先说明影响范围、风险和验证方式。
4. 小改动直接做；新功能、多模块改动、迁移、AI 流程、核心 UI 改造先写紧凑 spec 或 OpenSpec tasks。

## Protected Data

不要删除、覆盖、提交或推送以下内容，除非用户明确要求且后果已说明：

- `database.sqlite`、生产/本地数据库文件。
- 上传文件、用户生成内容和运行时数据。
- `.env`、密钥、环境配置。
- 本地日志、临时调试文件、Playwright 截图、测试输出。
- 未跟踪文件，尤其是可能包含素材、数据或交付物的文件。

用户说“推送但别推送数据”时，只推代码和必要静态资产。

## UI Quality

改动涉及 UI、交互、样式、弹层、响应式布局时：

- 同时检查桌面端和移动端。
- 同时关注白天/夜间主题。
- 检查文字溢出、重叠、遮挡、按钮状态、logo 对比度、背景、表单、搜索、筛选和导航。
- 当把原本半屏/列表里的内容改成全屏翻页、沉浸页或 `100svh` section 时，必须把每个 section 当成独立页面重做构图：重定信息层级、留白、组件尺寸、视觉锚点和操作区，不要只套 snap 容器或简单压缩到不溢出。
- 不随手添加解释性废话、随机标签、占位式文案。
- 动效要顺滑，不能明显拖慢页面。

## AI And Taxonomy

涉及活动分类、推荐、解析、AI 大脑、微信解析、学院/组织/社团/主办方等标准数据时：

- 优先使用 `axton-ai-standardization`。
- AI 输出必须尽量包含标准 ID、标签、置信度、候选项和人工校正入口。
- 迁移脚本必须先支持 dry-run 和报告，再考虑写入。
- 不要让旧标签体系和新标准分类无 adapter 混用。

## Verification

提交或推送前，必须运行与改动直接相关的检查，并在汇报中说明结果。

最低要求：

- 前端文件改动：至少运行相关 `eslint`，或说明为什么无法运行。
- 新增/修改测试：至少运行对应测试文件。
- 构建或路由风险：运行 `npm run build` 或对应更小检查。
- UI / 布局风险：用浏览器或 Playwright 检查关键桌面端和移动端路径。
- AI 助手风险：优先运行 `npm run check:ai-assistant` 或相关服务端检查。
- OpenSpec 相关变更：运行 `npm run openspec:validate` 或对应 change validate。

推荐命令：

- `npm run lint`
- `npx eslint <changed-files...>`
- `npm run build`
- `npm run test:e2e:smoke`
- `npx playwright test e2e/<relevant-spec>.js --project=chromium`
- `npm run check:ai-assistant`

## Do Not Commit

- 测试失败但未说明风险的改动。
- 明知没有验证移动端，却提交响应式 UI 改动。
- 数据库、上传文件、`.env`、本地日志、临时截图、无关生成物。
- 无关 lockfile 噪音、临时调试文件。
